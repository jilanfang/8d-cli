# risk-thinking — 基于风险的思维（ISO 9001:2015 第 6.1 条）

## 行业背景

基于风险的思维是 ISO 9001:2015 和 2008 版之间最核心的哲学转变。2008 版有"预防措施"这个独立条款（8.5.3），但多数组织只是在程序文件里写几段话应付审核。2015 版取消了预防措施的独立条款，把风险思维嵌入到 QMS 的每个环节——第 6.1 条（应对风险和机遇的措施）是总纲，但"风险"这个词在标准里出现了 48 次。

对 8D 的实际影响：
- 纠正措施本身也是变更，每次变更都该做风险评估
- 不能解决一个问题、引入三个新问题
- 措施的力度应该和风险匹配（高风险问题 → 消除级措施；低风险 → 检测级也可以接受）
- 不只有负面风险——还要看纠正措施会不会带来机遇（比如改进了一个工序，顺便提高了产能）

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 3-4 个向用户提问。

1. **措施的附带风险**："你选的这个措施——比如改回流焊温度曲线——会不会影响旁边的特性？会不会影响上游来料的兼容性？会不会影响后续组装工序？列出至少 2 个可能被连带影响的地方。"

2. **控制升级的风险**："加了新的检测——这个检测本身可靠吗？如果检测设备误报/漏报，会有什么后果？一个不可靠的检测加进去，和没加一样差。"

3. **引入新操作风险**："改了 SOP 后，操作员的学习曲线是什么？在新旧 SOP 切换期间，会不会因为不熟练产生新的不良？切换期的控制怎么保证？"

4. **供应商变更风险**："如果措施涉及供应商端变更——供应商有变更控制流程吗？他们的变更验证你能不能看到？供应商变更后你的来料检验能检出问题吗？"

5. **不行动的风险**："反过来问——如果因为担心新风险而不实施措施，保持现状的风险是什么？这个问题复发的概率和后果，vs 新措施引入风险的概率和后果，哪个更严重？"

## S2 | 第一性原理收敛

追问链 —— 三层风险判断。

**Layer 1 — 措施自身风险**
- "这个措施改变了什么输入/活动/输出？用一个 FMEA 思维——新的失效模式是什么？严重度？发生率？探测度？"
- "如果有功能安全要求（比如安全PLC控制的功能），这个措施改动了安全相关部分吗？需要做安全评估吗？"

**Layer 2 — 交互风险（最关键的一层）**
- "这个工序的输出是哪个工序的输入？改了这里，下游的输入假设是否还成立？你问过下游工序的工程师吗？"
- "这个参数改完，设备还用在其他产品上吗？如果设备是共享的——你对其他产品做了交叉验证吗？还是一共只测了这个产品？"

**Layer 3 — 系统风险**
- "如果这个措施使某个控制点更严格了（比如本来的 ±2mm 改成 ±1mm），有没有之前合格、新标准下不合格的产品还在客户那里？在途？在库存？"
- "如果这个措施让产能降低了、节拍变慢了、成本上升了——这些运营压力会不会导致其他地方偷工减料？"

## S3 | 反事实挑战

1. "假设一年后这个问题复发了——你回头看，最可能是因为什么当时没想到的东西？是哪个交互效应、哪个边界条件、哪个人的变更？"
2. "如果你必须在一个月内发现这个措施引入的新问题——你会盯哪个指标？"
3. "如果有人说'这个措施没有新风险，我们做过评估了'——你用什么方法验证这个评估不是走形式？风险评估是谁做的？有没有独立评审？"

## S4 | 答案评分

评分维度：**风险识别广度**、**交互效应分析**、**预防控制设计**、**风险接受合理性**

| 等级 | 标准 |
|------|------|
| Insufficient | 未考虑措施的风险；认为"新措施不会有问题"；没有列出任何被影响的特性/过程/接口 |
| Basic | 识别了至少 2 个连带影响点；对可能新引入的失效模式做了初步描述；有变更影响分析记录 |
| Excellent | 对每个连带影响点做了 FMEA 风险评估；用 SIPOC 或过程图标注了变更影响的上下游接口；下游工序的工程师确认了输入假设仍然成立；对共享设备/共享物料做了交叉验证；有风险接受记录和审批 |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| "措施通过了评审，没有风险" | 通过什么方法确认没风险？说清楚方法而非结论 |
| 只看自己工序不管上下游 | 至少审查直接上游和直接下游两个接口 |
| 对共享设备只验证一个产品 | 所有使用该设备的产品都要做交叉验证或书面说明为什么不需要 |
| 风险评估只有质量部签字 | 制造工程/生产/设备——三个职能必须有独立意见 |
| 风险接受没有管理层授权 | 高风险接受需要管理层签字；中风险需要 QA Manager 签字 |
