# IATF 16949 Profile Overlay — D0-D8 Augmentation

> Agent: 当你检测到用户处于汽车行业上下文（OEM投诉、PPAP涉及、特殊特性标注、Tier N供应商管理等），在基础 8d-guru 的每个 D 上叠加以下要求。每条要求标注了 IATF 16949 条款号以供追溯。

## 全局规则（适用所有 D）

1. **时效追踪**：任何 OEM 投诉，从"问题确认时间"开始计时。记录 timestamp。见 [oem-timelines.md](references/oem-timelines.md)。
2. **CSR 优先**：每个 D 的第一问追加："你的客户对这个 D 有什么特殊格式/时效/审批要求？"见 [csr-check prompt](prompts/csr-check.md)。
3. **特殊特性贯穿**：如果涉案产品有 CC/SC/HIC，所有分析、措施、验证必须特殊标注其影响。见 [special-chars prompt](prompts/special-chars.md)。
4. **条款责任人**：D1 RACI 中必须标注以下条款的 Owner：10.2.3（问题解决）、10.2.4（防错）、10.2.6（客户投诉）、8.4.2.4.1（供应商管理）。

## Hard Gates — IATF 16949 阻断规则

基础 8d-guru 的 gates（`docs/gates.md`）和 ISO 9001 profile 的 gates（`profiles/iso9001/overlay.md` 中的 Hard Gates 段）是基础阻断。以下 gates 是 IATF 16949 **汽车行业特有**的阻断——不满足时，即使基础 gate 和 ISO 9001 gate 通过，IATF gate 也必须拦住。

### Gate D0 → D1：启动阻断

| # | 阻断条件 | 严重性 | Agent 行为 |
|---|---------|--------|-----------|
| G0-I1 | 涉及 CC 特性但未启动最高级 8D（安全/法规相关→立即升级） | 🔴 BLOCK | "IATF 16949 8.3.3.3：涉及 CC（▽）特性的问题自动触发最高级响应。不能降级处理。立即通知管理层+客户 SQE。" |
| G0-I2 | 安全相关投诉未在 24 小时内通知客户 SQE | 🔴 BLOCK | "IATF 16949 10.2.6：安全相关投诉必须立即通知客户。24 小时内未通知客户→不符合 CSR。" |
| G0-I3 | OEM 初始回应 deadline 未知（未查 oem-timelines.md） | 🟡 WARN | 允许进入 D1，但必须在 D1 交付物中标注时限并启动倒计时 |

### Gate D1 → D2：团队阻断

| # | 阻断条件 | 严重性 | Agent 行为 |
|---|---------|--------|-----------|
| G1-I1 | 团队缺少设计工程师（如组织有设计责任）或缺少制造工程师 | 🔴 BLOCK | "IATF 16949 10.2.3：交叉职能团队必须含 Design（如有设计责任）+ Mfg + Quality。缺人的原因必须书面记录。如果设计责任在客户→标注并引用客户设计文件。" |
| G1-I2 | 客户 SQE 未被通知（涉及 CC 或安全特性时） | 🔴 BLOCK | "CC/安全特性必须通知客户 SQE。客户是否需要参与 D4/D6 审查？在 D1 就确认这一点。" |
| G1-I3 | RACI 中未标注 IATF 条款 Owner（10.2.3 问题解决/10.2.4 防错） | 🟡 WARN | 允许进入 D2，但必须在 D3 前补齐 |

### Gate D2 → D3：定义阻断

| # | 阻断条件 | 严重性 | Agent 行为 |
|---|---------|--------|-----------|
| G2-I1 | 涉及特殊特性（CC/SC/HIC）但图纸-PFMEA-CP 特殊特性链未闭合检查 | 🔴 BLOCK | "IATF 16949 8.3.3.3：特殊特性识别链必须闭合。图纸标了 CC→PFMEA 对应行必须有→CP 对应工序必须用对应控制方法。链断裂→D2 不完整。" |
| G2-I2 | PPAP 状态未知（哪版 PSW？Interim 还是 Full？Safe Launch 有效期中？） | 🔴 BLOCK | "IATF 16949 8.3.4.4：PSW 状态决定了后续 D5 PCA 是否需要重新提交 PPAP。先在 D2 搞清楚 PSW 状态。" |
| G2-I3 | Stellantis 客户且 NTF 未启动（退件未复现失效） | 🟡 WARN | "Stellantis CSR 强制 NTF 分析。其他 OEM 按 CSR 要求。" |

### Gate D3 → D4：围堵阻断

| # | 阻断条件 | 严重性 | Agent 行为 |
|---|---------|--------|-----------|
| G3-I1 | 24 小时初始回应 deadline 已过但客户未收到 D1-D3 | 🔴 BLOCK | "OEM CSR 要求 24 小时内初始回应。已超时→立即向客户 SQE 报告状态和原因。不能在没有通知客户的情况下继续内部分析。" |
| G3-I2 | 48 小时围堵计划未提交客户（OEM CSR 要求） | 🔴 BLOCK | "围堵计划必须在 48 小时内提交客户。先完成提交，再继续 D4。D3 是 D4 的前置条件而非并行任务。" |
| G3-I3 | 涉及 CC 特性但围堵措施是抽样而非 100% | 🔴 BLOCK | "IATF 16949 8.3.3.3：CC 特性不允许围堵抽样。必须是 100% 拦截。如果物理上无法 100%→书面记录+客户 SQE 批准+增强替代措施。" |
| G3-I4 | Safe Launch 期间但围堵未升级到 Safe Launch CP 级别 | 🔴 BLOCK | "APQP 3rd Ed.：Safe Launch 期间围堵需达到 Safe Launch 控制计划要求的增强频次和覆盖。常规围堵不足以满足 APQP 要求。" |
| G3-I5 | 特殊特性产品未在所有风险窗口 100% 标识和物理隔离 | 🟡 WARN | 允许推进但标识/隔离必须在 D4 完成前补齐 |

### Gate D4 → D5：根因阻断

| # | 阻断条件 | 严重性 | Agent 行为 |
|---|---------|--------|-----------|
| G4-I1 | 涉及 CC/SC 但根因未追溯到 PFMEA 对应行（工序编号+失效模式行+RPN） | 🔴 BLOCK | "IATF 16949 8.3.3.3：根因必须定位到 PFMEA 的对应工序和失效模式行。只说'发生了回流焊问题'不够——PFMEA 里是哪行？RPN 多少？" |
| G4-I2 | MRC 未回答"为什么 PFMEA/CP 没防止" | 🔴 BLOCK | "IATF 16949 10.2.3：系统性根因必须解释为什么现有预防/探测控制失效。PFMEA 当初是识别了但评低了，还是根本没识别？这个区别决定了 D7 预防措施的方向。" |
| G4-I3 | 4M 差异分析未覆盖从 OK 到 NOK 之间的所有变更（人/机/料/法） | 🔴 BLOCK | "IATF 16949 8.5.6.1：变更控制是 IATF 审核高频不符合项。如果根因可能与变更有关但没有查变更记录→根因可能漏掉最关键的事实。" |
| G4-I4 | 涉及供应商但供应商端 8D 未启动（D4 不完整） | 🟡 WARN | "IATF 16949 8.4.2.4.1：如果根因涉及供应商，供应商的 8D 也应在 D4-D5 推进。标注为待完成但不是阻断 D5 的条件——你自己的 D5 可以先进。" |

### Gate D5 → D6：措施阻断

| # | 阻断条件 | 严重性 | Agent 行为 |
|---|---------|--------|-----------|
| G5-I1 | PCA 未做 PPAP 影响评估（每项 PCA→是否需要重新提交 PSW→通知等级） | 🔴 BLOCK | "IATF 16949 8.3.4.4：任何涉及 fit/form/function/performance/durability 的变更必须评估 PPAP 影响。不经评估就直接实施的 PCA→在 IATF 审核中是不符合项。" |
| G5-I2 | 涉及 CC 特性但措施未达到 Poka-Yoke 消除级，且无书面技术理由+客户批准 | 🔴 BLOCK | "IATF 16949 10.2.4：CC 特性必须达到防错消除级。如果技术不可行→必须：a) 书面技术理由 b) 客户 CSR 批准 c) 增强替代控制方法 d) 增强验证频次。缺一不可。" |
| G5-I3 | PCA 需要客户批准（CSR 要求）但尚未获得 | 🔴 BLOCK | "CSR：部分 OEM 要求 PCA 在实施前必须经客户批准。检查客户的 CSR 文件——如果要求预批准但没拿到→先获得批准再实施。" |
| G5-I4 | 供应商端措施未追踪闭环 | 🟡 WARN | "IATF 16949 8.4.2.4.1：供应商的措施必须有追踪时间线和验证节点。标注为待完成但必须推进。" |
| G5-I5 | PFMEA/CP/SOP/图纸四文件未同步更新或有文件遗漏 | 🔴 BLOCK | "IATF 16949 10.2.3：至少 PFMEA 和 CP 必须同步更新。只改 SOP 不动 PFMEA/CP 是 IATF 审核的高频发现。" |

### Gate D6 → D7：验证阻断

| # | 阻断条件 | 严重性 | Agent 行为 |
|---|---------|--------|-----------|
| G6-I1 | 涉及 SC 特性但 Cpk < 1.33，或涉及 CC 特性但 Ppk < 1.67 | 🔴 BLOCK | "IATF 16949 8.3.3.3/SPC 手册：SC → Cpk ≥ 1.33 强制；CC → Ppk ≥ 1.67 强制。不达标→能力不足→措施不够。回到 D5 升级措施。" |
| G6-I2 | 防错装置未做 Red Rabbit 测试（IATF 10.2.4 强制） | 🔴 BLOCK | "IATF 16949 10.2.4：防错方法的有效性必须用挑战件（Red Rabbit）测试。这是强制项，不是可选项。" |
| G6-I3 | Safe Launch 期间但验证未覆盖 90 天增强监控计划 | 🔴 BLOCK | "APQP 3rd Ed.：Safe Launch 期间 D6 验证必须覆盖 90 天增强监控的频次和范围。常规验证不够。" |
| G6-I4 | LPA 未覆盖新增控制点 | 🟡 WARN | "IATF 16949 9.2.2.3：LPA 应覆盖新的控制措施。第一轮 LPA 结果进入 D7。" |

### Gate D7 → D8：预防阻断

| # | 阻断条件 | 严重性 | Agent 行为 |
|---|---------|--------|-----------|
| G7-I1 | PFMEA RPN 无变化或失效模式无新增——D7 的核心任务没完成 | 🔴 BLOCK | "IATF 16949 10.3.1：PFMEA 必须更新并降低 RPN。如果 RPN 没变→你的措施没有改变风险状态。检讨措施是否足够。" |
| G7-I2 | 供应商 8D 未闭环（你的 D8 不能独立于供应商的 D8） | 🔴 BLOCK | "IATF 16949 8.4.2.4.1：如果根因涉及供应商，供应商的 8D 必须闭环或至少 D6 验证数据已确认。你和供应商的 8D 是一个整体。" |
| G7-I3 | Lessons Learned 未进入 APQP 检查清单和新项目阶段评审 | 🟡 WARN | 建议结案但标注为必须完成的后续行动 |

### Gate D8：结案阻断

| # | 阻断条件 | 严重性 | Agent 行为 |
|---|---------|--------|-----------|
| G8-I1 | 客户未在 8D 报告上签署确认（CSR 要求时） | 🔴 BLOCK | "IATF 16949 10.2.6：部分 OEM 要求在报告上正式签核。需要签署但未签→不能内部结案。" |

| G8-I4 | 最终 8D 报告未通过客户指定门户系统提交（含所有附件） | 🔴 BLOCK | "CSR：通过邮件发 PDF 不算正式提交。必须通过 Amadeus/SQP/GQTS/NCT 等系统提交。" |

## 证据标准 — IATF 16949 审核级

IATF 16949 审核（包括第三方认证审核和 OEM CSR 审核）对证据要求比 ISO 9001 更严。以下汽车行业特化的证据标准：

### D/TLD 产品特殊要求

如果产品是 D/TLD（Documentation/Technical Liability Documentation，如 BMW 的 D/TLD、VW 的 D/TLD）：
- **所有 8D 相关记录保留 15 年**（从 EOP 起算）
- **5-Why 工作纸必须手写签名**（可扫描电子化但不能只有电子签名）
- **变更验证必须有边界样品（红/蓝/金样品）的照片证据**

### 逐 D 的证据标准补充

| D 步骤 | 汽车行业特化证据要求 |
|--------|-------------------|
| D0 | ERA 记录含 OEM ticket ID；安全特性投诉→客户 SQE 通知记录（邮件+电话时间） |
| D2 | 特殊特性清单+图纸符号照片；PPAP PSW 副本（或 PPAP 状态截图）；NTF 分析报告（如适用） |
| D3 | 24h/48h CSR 回应副本；Safe Launch CP（如适用）；100% 检查的数据记录（CC 特性） |
| D4 | PFMEA 对应页面（工序号+失效模式行高亮）；Red Rabbit 反证记录；4M 变更矩阵；供应商端 8D 的 D4 状态 |
| D5 | PPAP 影响评估表；客户（CSR）批准记录（如需要）；防错验证计划；四文件更新清单+对应版本号 |
| D6 | Cpk/Ppk 原始数据（非汇总）+量具校准记录；Red Rabbit 全部测试记录（日期/插入故障/结果/照片）；SPC 前后对比趋势图；Safe Launch 90 天监控数据（如适用） |
| D7 | PFMEA RPN 修订前后对比表；供应商端 8D 的 D6/D8 证据；LPA 包含新控制项的证据；APQP 检查清单更新记录 |
| D8 | 客户签核记录（或系统关闭截图）；PPAP 新 PSW 或状态确认；Safe Launch 退出记录（如适用）；供应商端 D8 签署记录 |

## 审计就绪检查（IATF 16949 Audit Readiness）

Agent 在 D8 S5 时，在 ISO 9001 审计就绪检查的基础上，加查以下 IATF 特化项：

```
□ 8.3.3.3 特殊特性链闭合：图纸→DFMEA→PFMEA→CP→SOP→检验规范，每个环节符号一致？
□ 8.3.4.4 PPAP 闭环：若 PCA 触发 PPAP→新 PSW 已批准或已提交？
□ 10.2.4 防错验证：每项防错有 Red Rabbit 测试记录？防错装置验证计划已定义？
□ 10.2.3 三轨根因：Occurrence + Non-Detection + Systemic 三个独立 5-Why 链完整？
□ CSR 合规：OEM 时限满足？客户格式/模板/系统要求满足？CSR 特殊条款（D/TLD/特殊特性符号）遵循？
□ 8.4.2.4.1 供应商闭环：涉及供应商的全部 8D 已闭环或 D6 已验证？
□ APQP Safe Launch：Safe Launch 期→退出条件满足？不满足→延长已批准？
□ 9.2.2.3 LPA 覆盖：新控制措施已进入 LPA 检查清单？第一轮 LPA 结果？
□ 10.3.1 PFMEA 更新：受影响行 RPN 降低？新失效模式已添加？
```

如果任何一个检查项未通过 → 不关闭 D8 gate → 必须补足或正式记录残余风险接受+客户 SQE 知悉。
