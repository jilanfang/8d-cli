# d0-q3 — 严重度与启动判断

## 行业背景

不是每个质量事件都需要完整 8D。ASQ 和 Ford 8D 手册都规定了启动门槛：安全相关事件、重大功能失效、重复发生的不良、客户正式投诉（非一般反馈）、法规合规风险 —— 满足任一即应启动 8D。IATF 16949 要求组织定义问题升级准则（escalation criteria），并在 D0 阶段做出"是否需要完整 8D"的书面判断。错误的启动决策有两个方向，都致命：该启动而不启动，问题被日常流程掩盖，最终扩大为客户索赔或召回；不该启动而启动，组织资源被稀释，真正的重大风险得不到聚焦。判断必须基于多维度风险矩阵，不能凭感觉。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **安全维度**："这个问题有没有涉及人身安全？有没有可能导致人员受伤？即使概率很低，潜在的伤害后果是什么？有没有安全相关的行业标准或法规要求？"
2. **功能维度**："产品的主要功能是否完全丧失？还是部分功能降级？客户还能不能继续使用？有没有临时规避方法？"
3. **法规维度**："这个问题涉不涉及强制性认证（CCC/UL/CE/CSA）？涉不涉及环保法规（RoHS/REACH）？涉不涉及行业监管（医疗/FCC/汽车安全）？不符合法规的后果是什么？"
4. **规模维度**："问题产品有多少数量？涉及多少客户？是单件还是批次性问题？如果还没有统计数据，你最悲观的估计是多少？"
5. **客户影响维度**："客户有没有因为这个事情停线？有没有提到索赔？有没有说要切换供应商？客户方的哪个级别的人在关注这件事？"

## S2 | 第一性原理收敛

追问链 —— 从表面判断逐层深入：

**Layer 1 — 风险具体化**
- "你说'有安全风险'—— 具体是什么风险？最坏会发生什么？发生的概率大概是多少？有没有类似产品的历史事故记录？"
- "你说'客户很生气'—— 客户具体说了什么？有没有正式的书面投诉或索赔函？"

**Layer 2 — 风险量级交叉验证**
- "你说'只影响了几台'—— 这个数量是怎么得出的？查了所有库存和在制品吗？还是只查了客户退回的？"
- "你说'功能失效但客户能接受'—— 这个判断是客户说的还是我们猜的？有没有客户的书面确认？"

**Layer 3 — 资源配置判断**
- "如果启动完整 8D，你估计需要几个人的多长时间？现在有谁可以投入？"
- "如果不启动 8D，你准备用什么流程来处理？日常纠正预防流程够不够？"

## S3 | 反事实挑战

1. "如果你的判断是'需要完整 8D'，什么情况下这个判断是错的 —— 也就是说，什么证据会让你认为这个问题可以用更轻的流程处理？反过来，如果你判断'不需要完整 8D'，什么新发现会让你改变主意？"
2. "目前最不确定的维度是哪一个（安全/功能/法规/规模/客户影响）？如果这个维度的实际情况比你现在假设的严重 10 倍，你的判断会变吗？"

## S4 | 答案评分

评分维度：**多维度覆盖**、**依据充分性**、**决策透明性**

| 等级 | 标准 |
|------|------|
| Insufficient | 只考虑了单一维度；判断无依据（"我觉得严重/不严重"）；风险矩阵缺失 |
| Basic | 安全/功能/法规/规模/客户影响五个维度均已评估；每个维度有事实或数据支撑；启动/不启动判断有书面理由 |
| Excellent | 每个维度给出最坏情况和概率估计；判断依据可追溯（邮件/数据/标准条文）；标明了最大不确定性在哪一维度；如果暂时无法判断，清晰地列出了还缺什么信息 |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 凭感觉判断"严重/不严重" | 五个维度必须有事实或合理假设支撑，假设要标注 |
| 只看客户情绪不看客观风险 | 客户没投诉但安全风险真实存在的，必须启动 8D |
| 用"数量少"作为唯一不启动理由 | 安全/法规维度的一件失效也需要完整 8D |
| 在没有评估的情况下列出判断 | 判断理由必须逐维度引用事实或数据来源 |
| 启动判断模棱两可 | 明确写 yes/no。不确定时写 no + 触发重新评估的条件 |
