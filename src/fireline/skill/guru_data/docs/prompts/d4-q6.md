# d4-q6 -- 控制点判断 + MRC(管理根因)

## 行业背景

MAHLE 8D 要求技术根因之上必须再追 MRC -- Managerial Root Cause。技术根因告诉你"什么失效了"，MRC 告诉你"为什么管理系统允许它发生并流出"。ASQ/BSH 观察到的规律:同类型问题反复发生的企业，技术根因几乎每次都找到了，但 MRC 没触及。结果是换一个产品、换一条线，同样的管理漏洞以不同的技术面貌重现。

控制点从浅到深六层:设计、制造过程、检测与放行、供应链、配置与变更、组织与管理。

## S1 | 发散探索 -- 多角度触发器

1. d4-q3 的 Occurrence 根因对应哪个控制点?属于六层中哪一层?是单控制点失守还是多个?
2. d4-q4 的 Escape Point 对应哪个控制点?和 Occurrence 的控制点是同一个吗?
3. 两个控制点失守是独立事件还是共享同一个系统漏洞?
4. 这个控制点谁设计的?依据什么标准?设计时考虑了哪些风险?有没有 FMEA?
5. KPI 激励了什么?资源有没有限制?信息有没有断裂?

## S2 | 第一性原理收敛 -- MRC 四要素

**管理系统**:
- "这个控制点有没有被 QMS 覆盖?文件化了吗?"
- "谁来监督有效性?最近一次监督发现了什么?"
- "管理评审有没有涉及这个领域?有没有提出过风险?"

**业务流程**:
- "这个控制点依赖什么上游输入?输入质量如何?向下游传递什么?下游有没有反馈不准?"
- "发现这个控制点有问题，有什么升级渠道?上次升级是什么时候?被处理了吗?"

**人员**:
- "涉及人员是否具备技能和授权?绩效评估是否奖励了正确行为?"
- "有没有无意中惩罚了提出问题的人?关键人员离岗时知识怎么传递?"

**组织**:
- "责任部门是谁?和其他部门的接口有没有灰色地带?"
- "有没有'看得见问题的人没权力改，有权力改的人看不到问题'?"
- "最近有没有组织调整削弱了这个控制点?"

## S3 | 反事实挑战

1. 如果技术根因对了但 MRC 找错了 -- D5 全部照做，同一个管理漏洞会在另一个产品/工序上再次暴露吗?
2. 这个控制点失效是"一次性意外"还是"系统设计保证它迟早会失效"?有没有历史上类似失效证据?

## S4 | 答案评分

| 维度 | Insufficient | Basic | Excellent |
|------|-------------|-------|-----------|
| 控制点定位 | "质量管理"这种模糊标签 | 具体控制点+对应层次 | 具体控制点+层次+为什么是这层不是更深/更浅 |
| Occurrence/Escape 关联 | 未区分两个控制点 | 分别识别了两个控制点 | 分别识别+分析了是否共同系统漏洞 |
| MRC 深度 | 停在"加强管理" | 触及管理系统或流程缺口 | 覆盖四要素中至少两个的具体缺口 |
| 可行动性 | MRC 太模糊无法对应 D7 | MRC 可对应改进行动 | 能区分一次性修补 vs 系统性改进 |

D4 铁律:控制点写成"质量控制""过程管控"而非具体工序/参数，或 MRC 写成"管理不到位"，一律 Insufficient。

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 控制点写万能标签 | 追问:哪个工序、哪个参数、哪个检查项、哪个文件章节? |
| MRC 写"管理不到位/执行力不足" | 追问:哪个系统要素缺失?文件/审核/资源/激励/信息/权力? |
| Occurrence 和 Escape 控制点混在一起 | 必须分开写，即使共享同一漏洞 |
| 只写技术层不写管理层 | 追问:"管理系统为什么允许它存在?" |
| 跳过多因交互中控制点依赖 | 写清楚:A 失效因为依赖 B，B 本身也有问题 |
