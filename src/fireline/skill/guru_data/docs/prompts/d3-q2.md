# d3-q2 — 在途产品围堵

## 行业背景

IATF 16949 第 8.7.1 条要求对不合格输出的围堵范围包括所有已发货但未到达客户的产品。AIAG 的 CQI-20（Effective Problem Solving）明确指出，在途产品是围堵中最容易被跳过的窗口——因为物流信息分散在多个系统和部门，没有人主动去查。Ford 的 Q1 审核中，在途围堵缺失是常见的扣分项。ASQ 强调：围堵的完备性取决于"从原材料到客户手中的每一站都有人负责"，在途段的责任归属最容易出现真空。

## S1 | 发散探索 — 多角度触发器

1. **有多少产品正在运输途中？** 你是查了 ERP 发货记录、物流系统、还是问了物流公司？这些数据互相一致吗？
2. **在途产品能不能拦截？** 是海运、陆运还是空运？货到哪个节点了——还在你的仓库门口、在港口、在海上、还是已经到了客户所在城市的分拨中心？
3. **如果拦截成功，产品怎么处置？** 拉回哪里？谁负责接收？接收后放入哪个区域——隔离区、待检区、还是直接上筛选线？
4. **如果拦截不成功（船已经出海、货已经到客户分拨中心），你的 Plan B 是什么？** 提前通知客户拒收吗？客户那边有没有对接人知道这批货可能有问题？
5. **有没有不在常规系统里的"灰色在途"？** 比如临时调拨、样品寄送、售后备件、展会后回运的产品——这些往往不在发货台账里，但它们是真实存在的在途品。

## S2 | 第一性原理收敛

追问链（对涉及在途的窗口逐条追问）：

- 你说的"已经联系物流公司拦截了"，物流公司回复你的拦截确认单在哪里？拦截指令是谁发的、几点发的？拦截成功之前货会不会被转运到下一站？
- 拦截下来的产品，你打算怎么处理？拉回工厂后直接上筛选线，还是先放入待检区？谁来判定这批产品放哪里——有没有书面指令？
- 如果拦截失败、货已经到客户仓库门口了但还没签收，客户会不会直接入库？你有什么机制确保客户不入库先用？
- 在途产品的数量和你的发货记录对得上吗？有没有可能漏了某一批发货？你是怎么确认"100%在途受影响数量"的——手工核对、系统比对、还是两条线交叉验证？

## S3 | 反事实挑战

"如果你的在途拦截失败了——物流公司说找不到那批货、或者货已经在客户分拨中心入库了——客户手里突然多了一批你还没有围堵过的可疑品。到那个时候，你的围堵措施是提前通知了客户'可能有问题别用'，还是客户对此一无所知？如果是一无所知，这批货会直接上产线吗？"

## S4 | 答案评分

| 维度 | 不通过（0分） | 合格（1分） | 深入（2分） |
| :------ | :--------- | :------- | :------- |
| **物流信息完整性** | "货在路上，不知道具体在哪"，没有运单号 | 有运单号+当前位置+预计到达时间 | 有运单号+位置+ETA+物流公司联系人+拦截可行性判断 |
| **拦截方案可行性** | "已通知物流"但没有确认反馈 | 有拦截指令+物流确认+备选方案 | 有拦截指令+物流确认+接收方案+失败后的客户预警机制 |
| **数量准确性** | 只有一个大概数量 | 发货记录精确数量+系统截图 | 发货记录+灰色在途排查+交叉验证+精确到件数的清单 |

## 纠偏原则 / 禁止项

- **禁止跳过"不知道该查哪些系统"这一步就下结论。** 在途产品的信息来源通常分散在 ERP、WMS、TMS、物流公司系统、甚至 Excel 台账里。只说"查了系统"但说不出查了哪个系统，等于没查。
- **禁止把"通知了物流公司"当作围堵完成。** 通知不等于拦截，拦截不等于完成。必须追到拦截确认单、货物物理位置的确认、以及拦截后产品的处置方案。
- **禁止遗漏非标准发货。** 样品、售后备件、展会返回品、临时调拨——这些不经过标准发货流程的产品，最容易跳过在途围堵。
- **禁止在没有核实的情况下说"在途不涉及"。** 如果发货日期在问题时间段之前且还没有签收，这批货就是在途。要从发货台账出发，按运单逐票核实——不是因为"我觉得应该没发货"就不涉及。
