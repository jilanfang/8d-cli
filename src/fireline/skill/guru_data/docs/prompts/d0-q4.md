# d0-q4 — 紧急处置措施（ERA）

## 行业背景

ERA（Emergency Response Action）是 8D 中最容易被混淆的概念之一。Ford 8D 手册明确定义：ERA 是在根因尚不明确的紧急情况下，为防止问题扩大而采取的立即行动，发生在 D0 阶段。它与 D3 围堵措施（Interim Containment Action, ICA）的区别在于：ERA 是应急反应、时间驱动、范围优先（先控制再精确），ICA 是计划性隔离、数据驱动、范围精确（基于对问题的初步理解）。AIAG CQI-20 强调 ERA 必须在 24 小时内启动并文档化。常见错误：把 ERA 和 ICA 混为一谈，或者根本不知道 ERA 是什么、做了什么也没记录。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **已执行动作**："从发现这个问题到现在，你们已经做了什么？不管是正式的还是非正式的，只要是针对这个问题的控制动作，都说出来。"
2. **时间线**："每个动作是什么时候做的？谁做的？动作生效的时间点是什么？从发现问题到采取第一个动作，中间过了多久？"
3. **范围判断**："当时的控制范围是怎么定的？全检了所有库存？只检查了客户退回的？停用了某个设备？切换了某个供应商的料？怎么决定的范围边界？"
4. **效果验证**："这些动作做了以后，问题有没有继续发生？你怎么确认动作生效了？用什么指标判断的？"
5. **沟通记录**："ERA 有没有通知到客户的对应接口人？什么时候通知的？以什么形式（电话/邮件/会议）？有没有书面记录？"

## S2 | 第一性原理收敛

追问链 —— 从表面记录逐层深入：

**Layer 1 — 动作具体化**
- "你说'全检了库存'—— 全检了多少件？用了什么检验方法？检验标准是什么？谁检的？检出几件不良？"
- "你说'停用了这台设备'—— 设备编号多少？什么时候停的？停了之后有没有影响到其他产品的生产？"

**Layer 2 — ERA vs ICA 区分**
- "你做的这些动作是在知道根因之前做的还是之后做的？如果是之前做的，你做这些动作时是基于什么判断？如果是之后做的，那这不是 ERA，应该归到 D3。"
- "这些动作有没有明确的撤销条件？也就是说，什么条件下你会停止这些紧急措施？"

**Layer 3 — 缺口识别**
- "除了你现在做了的这些，有没有你觉得应该做但还没做的事？是什么阻止了你去做？"
- "有没有人提出过一个紧急动作但被否决了？否决的理由是什么？这个理由现在还成立吗？"

## S3 | 反事实挑战

1. "假设你采取的某个紧急措施实际上没有生效 —— 问题在被'控制'的状态下仍在发生 —— 你什么时候会意识到这一点？通过什么信号？"
2. "如果你的 ERA 范围定小了（比如只围堵了库存但问题出在在制品），现在回过头来觉得最可能遗漏的窗口是哪一个？"

## S4 | 答案评分

评分维度：**时效性**、**记录完整性**、**范围合理性**

| 等级 | 标准 |
|------|------|
| Insufficient | 不清楚做了什么紧急措施；没有时间记录；没有责任人；ERA 和 ICA 混淆 |
| Basic | 每项措施有 what/who/when；有范围说明；有与 ICA 的区分意识；客户已通知或有通知计划 |
| Excellent | 每项措施有生效验证；明确了撤销条件；所有风险窗口（客户现场/在途/库存/WIP/上游）均已扫描；有书面通知记录；ERA 到 ICA 的过渡逻辑清晰 |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 把 D3 围堵措施当 ERA 写进来 | ERA 是根因不明前的应急反应，ICA 是有初步理解后的计划隔离。时间上和逻辑上都要区分 |
| ERA 没有时间记录 | 应急反应的时间窗口是关键审核点，每个动作必须有精确时间 |
| "我们没做什么特别的动作" 当答案 | 追问"那发生了什么？问题还在继续吗？还是自己好了？" |
| ERA 没有撤销条件 | 紧急措施不能永久化，必须明确什么时候退出或过渡到 ICA |
| 动作描述笼统（"加强了检验"） | 追问：检了多少、用什么方法、什么标准、谁检、检出多少 |
