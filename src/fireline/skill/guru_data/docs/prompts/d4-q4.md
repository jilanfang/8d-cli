# d4-q4 -- 流出原因(Escape Point)

## 行业背景

BSH 8D 将 Escape Point 与 Occurrence 并列为 D4 两大支柱。逻辑:问题发生了(第一个失败)，问题流出了(第二个独立失败)。两个失败可能共享根因，也可能完全不同。ASQ 的观点:如果 8D 只有 Occurrence 没有 Escape，说明只修好了"为什么发生"，没修好"为什么下次还会流出"。换一个失效模式，同样的检测漏洞还在。Escape Point 是组织的免疫系统漏洞 -- 它保护的是顾客。

## S1 | 发散探索 -- 多角度触发器

1. 从失效点往上游逐个检查:OQC、最终检验、过程检验、首件检验、来料检验 -- 每道关卡检查了什么项目?标准是什么?实际执行了什么?
2. 最早在哪个节点，这个失效模式**技术上可以被检测到**?不是"是否应该检出"，是"有没有能力检出"。
3. 在每个关卡:检验员依赖什么信息做判断?目视、设备读数还是自动判定?误判率和漏判率分别是多少?
4. 有没有人看到过异常但判断为"没问题所以放行"?那个判断的依据是什么?后来被推翻了吗?
5. 流出到客户的路径上有没有信息断裂?产线知道有问题但出货部门不知道?

## S2 | 第一性原理收敛

**检测原理追问**:
- "这个检测点的检测原理能不能可靠地捕获这个失效模式?原理上有盲区吗?"
- "检测设备的分辨率/精度/重复性是否足以识别这个失效模式的量级?"
- "检测是 100% 全检还是抽检?AQL 对这个失效模式和客户风险的适配性如何?"

**标准充分性追问**:
- "检测标准怎么来的?是否覆盖了这个失效模式?"
- "缺陷目录里有这个失效模式吗?不在里面的话，检验员能识别吗?"
- "如果标准本身没有覆盖该模式，检验员就算完美执行也拦不住 -- 这是系统设计问题。"

**人-系统边界追问**:
- "如果是人工判断，疲劳/训练/轮班对判断质量影响多大?"
- "检验员的判定被推翻后，有没有反馈渠道让他知道'之前的判断是错的'?"

## S3 | 反事实挑战

1. 如果你认定的 Escape Point 不是最早能检出的点 -- 往前走一个节点，那个节点有没有能力检出?为什么说它"不能"?
2. 假设所有设备和人员完美运作 -- 在这个理想条件下，这个失效还会流出吗?如果"会"，检测体系设计本身有盲区。

## S4 | 答案评分

| 维度 | Insufficient | Basic | Excellent |
|------|-------------|-------|-----------|
| 回溯完整度 | 只说了最终检验 | 覆盖主要关卡 | 来料→首件→过程→最终→出货，完整回溯 |
| 检出能力 | 只说"漏检"不解释 | 说明检测原理与失效模式匹配性 | 定量分析检测能力与失效模式量级的关系+盲区 |
| 最早检出点 | 未确定 | 确定了最早检出点 | 最早检出点+为什么该点控制设计存在盲区 |
| 标准审查 | 未审查标准 | 提到标准可能不充分 | 审查了标准来源/依据及与失效模式的覆盖度 |

D4 铁律:只说"检验漏检"但没有说明哪个检验、为什么漏、最早能检出的点在哪，一律 Insufficient。

## S5 | 输出格式

```json
{
  "escape_chain": [
    {
      "detection_point": "来料/首件/过程巡检/过程自检/成品/OQC/客户验收",
      "what_checked": "实际检查项目",
      "standard": "标准/规格/阈值",
      "did_catch": "yes/no",
      "why_escaped": "没检这个/标准太宽/方法不灵敏/误判/没执行"
    }
  ],
  "earliest_possible_detection": "技术上最早能检出的节点",
  "primary_escape_point": "最应该拦住但没拦住的节点",
  "escape_root_cause": "流出根因(独立于发生根因)",
  "detection_blind_spot": "检测体系设计盲区",
  "confidence": 0.0
}
```

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 只说"检验漏检"不展开 | 必须说:哪个检验?检什么?为什么漏?没检/检不出/检不准/标准错/人漏了? |
| 把 Occurrence 和 Escape 画等号 | 两条独立因果链，分别分析 |
| "最后一道检验没拦住"就停 | 继续:前面每一道能不能拦住?最早是哪一道? |
| "客户没发现所以没数据" | 客户验收是最后一道防线，前面都是你的责任 |
| 跳过了标准审查 | 标准没覆盖这个模式 = 系统设计问题，不是人的问题 |
