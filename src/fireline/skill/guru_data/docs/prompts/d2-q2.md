# d2-q2 — 厂内视角问题描述

## 行业背景

客户视角和内部视角的语言体系不同。客户说"没反应"，内部对应的是"MCU 未启动"还是"电源模块无输出"还是"连接器接触不良"？ASQ 8D 方法论要求将客户语言翻译为可测量的内部工程语言，这个翻译过程本身就是问题定义的关键步骤。如果翻译错误（比如把"不亮"直接翻译成"LED 坏"，而实际上是电源问题导致 LED 不亮），D4 根因分析从一开始就走错了方向。IATF 16949 的"客户特定要求"条款也隐含了这个逻辑 —— 组织必须有能力将客户的语言转化为内部的技术规范和控制参数。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **工程翻译**："客户说的'XXX'，在你们的工程语言里对应的是什么？这个翻译你们团队内部有没有不同意见？有没有人觉得客户说的可能不是那个意思？"
2. **工艺/工序映射**："这个问题在厂内对应哪道工序？哪个测试站？哪个检验点？这个工序/测试站有没有历史不良数据可以对比？"
3. **内部复现**："你在厂内试过复现这个问题吗？用同样的条件（电压/温度/负载/操作步骤），能不能让问题在内部出现？如果内部复现不了，你觉得原因是什么？"
4. **内部检测能力**："厂内的测试和检验能不能发现这个问题？如果能发现，为什么还是流出去了？如果不能发现，是测试方法不够还是根本没测这个项目？"
5. **内部已有记录**："在 MES/ERP/QMS 里，有没有和这个产品、这个工序、这个时期相关的异常记录或趋势变化？有没有人已经注意到了一些信号但没形成正式报告？"

## S2 | 第一性原理收敛

追问链 —— 从表面翻译逐层深入：

**Layer 1 — 翻译的准确性**
- "你说客户说的'异响'对应的是'轴承异常振动'—— 你怎么确认的？有没有拆解过客户退回的样品？还是根据客户录音判断的？"
- "内部不同职能的人对客户同一句话的理解可能不同：质量工程师怎么看？工艺工程师怎么看？设计工程师怎么看？他们的解释矛盾吗？"

**Layer 2 — 内部检测链**
- "从产品做完到出厂，经过了几道检验或测试？这些检验项目是什么？每一个检验项目的判定标准是什么？这个缺陷在哪个检验点应该被检出但实际上没有被检出？"
- "如果厂内做了全检但没发现，是全检方法有漏洞，还是全检数据被误判了，还是问题发生在全检之后（比如运输/存储）？"

**Layer 3 — 数据回溯**
- "能不能拉出过去 6 个月这个产品在对应工序/测试站的数据趋势？有没有在失效发生前后出现统计上的偏移（哪怕还在规格内）？"
- "有没有处理过看起来类似的内部不良但当时被当成偶发事件关了？那个案例的记录在哪？"

## S3 | 反事实挑战

1. "假设你们的'工程翻译'是错的 —— 客户描述的现象 A 实际上对应的是内部技术问题 B 而不是你们假设的 C —— 你在什么阶段、通过什么证据会发现自己错了？"
2. "如果问题可以在客户现场复现但厂内任何条件下都复现不了，这说明什么？可能的原因是什么？（环境差异？使用方式差异？批次差异？测量方法差异？老化时间差异？）"

## S4 | 答案评分

评分维度：**翻译准确性**、**内部可追溯性**、**复现状态**

| 等级 | 标准 |
|------|------|
| Insufficient | 工程翻译无依据（猜测）；不清楚内部对应工序/测试；没有尝试内部复现；没有查阅历史数据 |
| Basic | 工程翻译有物理依据；内部工序/测试站已定位；已尝试内部复现（成功或知道为什么失败）；查阅了相关历史数据 |
| Excellent | 多职能（质量/工艺/设计）对工程翻译达成一致；内部检测链路完整映射（哪道检/怎么判/为什么流出）；内部数据趋势已回溯；客户环境与内部环境的差异已系统对比 |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 一个人拍板工程翻译 | 至少质量/工艺/设计三个职能背靠背翻译，对比差异 |
| 不尝试内部复现就直接分析根因 | 复现是 D4 根因验证的基础，D2 就要开始准备复现条件 |
| 跳过厂内检测能力评估 | 流出原因（escape point）是 8D 的独立分析线，D2 就要建立基本判断 |
| 靠记忆判断"以前没有过" | 拉历史数据看趋势，概率性的偏移可能已经在发生 |
| 把内部翻译当唯一真理 | 始终保留"翻译可能不准"的假设，新证据出现时重新审视 |
