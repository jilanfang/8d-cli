# d1-q2 — 客户接口人

## 行业背景

客户接口人是 8D 团队中唯一对外发声的通道。ASQ 8D 方法论和 AIAG CQI-20 均明确：8D 过程中的所有客户沟通必须经由单一接口人，这是防止"多头对外、信息不一致"的基本纪律。接口人的核心职责不是回答问题，而是管理客户的期望和信任 —— 什么时候更新、更新什么内容、谁可以批准发布。IATF 16949 第 9.1.2.1 条要求客户满意度管理包括对投诉处理的沟通时效和一致性。一个弱的客户接口人会导致：不同人对客户说了不同版本的问题定义、客户从销售那里听到了质量没确认的说法、围堵措施没通知到位导致客户停线时措手不及。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **身份确认**："谁负责和客户沟通这个 8D 的一切信息？不是谁'也能联系客户'，而是谁是这个 8D 的唯一对外窗口？"
2. **沟通权限**："这个人在发任何对客户的更新之前，需要谁审批？审批链有几级？如果他觉得需要立即通知客户（比如发现安全风险），能不能跳过常规审批直接发？"
3. **沟通计划**："和客户沟通的频率和方式是什么？每周邮件更新？每次有进展就电话？正式报告多久发一次？这个计划客户同意了吗？"
4. **关系历史**："这个接口人和客户方的接口人之前有没有打过交道？关系怎么样？有没有信任基础？如果客户方接口人不好沟通，有什么备选策略？"
5. **危机沟通**："如果情况恶化（比如客户停线、媒体关注、监管介入），沟通升级路线是什么？谁对客户高层说话？说什么？"

## S2 | 第一性原理收敛

追问链 —— 从表面答案逐层深入：

**Layer 1 — 角色确认**
- "你说'销售经理负责和客户沟通'—— 这个销售经理懂不懂这个产品的技术？如果客户问技术细节，他能不能准确传达？还是需要技术角色参会？"
- "你说'我来跟客户说'—— 你跟客户说什么语言？中文还是英文？如果是英文报告，你写还是别人写好你翻译？"

**Layer 2 — 信息一致性**
- "除了你，还有谁会跟客户谈这件事？CEO？销售？FAE？你怎么确保他们从你这里拿到的是同一套说法？"
- "如果客户绕过你直接找了你们公司的另一个人（比如老板），那个人会不会回复？你准备怎么建立这个纪律？"

**Layer 3 — 沟通节奏**
- "客户有没有给过时间要求？比如'48 小时内给初步报告''7 天内给围堵方案'？这些时间要求你判断是否合理？如果做不到，你怎么跟客户谈？"
- "你上次跟客户沟通是什么时候？下次是什么时候？这个节奏客户满意吗？"

## S3 | 反事实挑战

1. "假设客户听了你的更新以后，理解的和你想传达的完全相反 —— 你觉得是哪个环节最容易出这种偏差？是语言问题、术语问题、还是你简化的东西刚好是客户最在乎的？"
2. "如果这个客户接口人也恰好是问题根因可能涉及的人（比如他就是负责这个产品设计的），客户发现了这个利益冲突，会怎么想？你怎么提前处理这个问题？"

## S4 | 答案评分

评分维度：**专一性**、**沟通计划性**、**能力匹配性**

| 等级 | 标准 |
|------|------|
| Insufficient | 没有明确接口人；多人同时和客户沟通；没有沟通计划；不了解客户的时间要求 |
| Basic | 接口人是具名个人；已确认单一窗口原则；有沟通频率和方式计划；知道客户方接口人是谁 |
| Excellent | 沟通计划经客户确认；有信息一致性纪律（内部其他人如何统一说法）；有危机沟通升级路线；接口人具备该产品/客户的基本技术知识；已评估利益冲突风险 |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| "销售顺便跟客户说" 当作沟通计划 | 8D 沟通必须是结构化、有计划、有记录的，不能"顺便" |
| 多头对外、各自跟客户说不同版本 | 所有客户沟通汇集到单一接口人发出 |
| 接口人不懂产品技术 | 接口人不需要做根因分析，但必须能准确传达技术信息和理解客户的技术问题 |
| 没有客户的沟通节奏要求 | 主动问客户期望的更新频率，不要按自己方便定 |
| 危机沟通无预案 | 安全/法规/停线级别的升级路线必须提前明确 |
