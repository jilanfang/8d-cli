# d2-q5 — 已知信息与缺失信息

## 行业背景

在 8D 的方法论体系中，承认"我们还不知道什么"比"我们知道什么"更能驱动有效的调查。ASQ 8D 指南和 AIAG CQI-20 都把信息缺口识别作为 D2 问题定义的标准步骤。一个常见的认知陷阱是：团队把已知信息、推断和假设混在一起，形成一个看似自洽但实际充满信息缺口的问题描述。Ford 的 Team Oriented Problem Solving (TOPS) 方法特别强调：问题定义的最后一步不是总结已知，而是列出"为了回答 D4 根因问题，我们还必须知道什么"。没有这个缺失信息清单，D3 围堵和 D4 根因分析的效率会大幅降低 —— 因为团队不知道应该去收集什么。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **已知盘点**："关于这个问题，现在你手上有什么信息？把所有你知道的列出来 —— 产品信息、现象、时间、数量、涉及人员、已有的检验/测试结果、客户的反馈。每一条都要写下来，不要因为你'觉得不重要'就跳过。"
2. **缺失识别**："如果你现在必须向一个完全不了解这件事的技术专家说清楚问题，你觉得你缺少什么信息他才能开始分析？不是'更多信息'这种废话，是具体条目。"
3. **获取路径**："你列出的这些缺失信息，每一条有没有明确的获取方法？去找谁？查什么系统？做什么测试？需要多久？有没有哪一条你觉得获取不了？障碍是什么？"
4. **信息优先级**："在这些缺失信息里，哪些是阻塞性的 —— 没有它下一步根本没法推进？哪些是锦上添花 —— 有了更好但不是必须的？"
5. **信息可靠度**："你已有的'已知信息'里，每条信息的可靠度如何？有没有信息来源不可靠但你一直按可靠在用的？（比如客户口头描述从未书面确认）"

## S2 | 第一性原理收敛

追问链 —— 从表面清单逐层深入：

**Layer 1 — 已知信息审查**
- "你说'已知产品在 2403 批次有问题'—— 你确定 2403 批次全部有问题，还是 2403 批次里的部分产品有问题？这个区分很重要。"
- "你说'测试报告显示电压正常'—— 你看过原始测试报告吗？什么时候测的？用什么设备测的？测试条件是什么？"

**Layer 2 — 缺失信息的切分**
- "你列的缺失信息里，哪些是 D2 阶段就需要的信息（问题边界相关），哪些是 D4 阶段需要的信息（根因分析相关）？前者现在就要拿，后者可以在 D3 围堵期间并行收集。"
- "对于阻塞性缺失信息，如果最理想的获取方法走不通，有没有 Plan B？比如如果供应商不配合提供批次追溯数据，有没有替代的数据来源？"

**Layer 3 — 未知的未知**
- "有没有一些信息是你觉得自己不应该不知道但实际上不知道的？—— 比如你不确定这个产品的设计规格是不是对的，或者你不确定这个工序有没有变更过。"
- "如果让一个完全不懂你们产品但擅长质量调查的人来看你的信息清单，他最可能问的一个'你们怎么连这都不知道'的问题是什么？"

## S3 | 反事实挑战

1. "假设你'已知信息'里最重要的一条其实是错的（比如你说可靠的测试数据实际上是测错了），这会导致你的整个问题定义往什么方向跑偏？你怎么保护自己不被这一条错误信息带歪？"
2. "如果你三个月后回头看今天列出的'缺失信息清单'，你觉得哪一条是你当时根本没想到但后来证明是关键的信息？—— 这个思考练习能帮你发现'未知的未知'。"

## S4 | 答案评分

评分维度：**完整性**、**优先级**、**获取可行性**

| 等级 | 标准 |
|------|------|
| Insufficient | 没有系统盘点已知信息；缺失信息写成"还需要调查""还需要分析"等空泛表述；没有获取方法；未区分 D2/D4 信息需求 |
| Basic | 已知信息清单化（≥10 条）；缺失信息具体（"缺 2401 批次的测试数据"而非"缺数据"）；每条缺失信息有获取方法或 Plan B；阻塞性信息已标出 |
| Excellent | 已知信息按可靠度分级（确认/推测/待验证）；缺失信息按优先级和阶段（D2/D4）分类；阻塞性信息有明确的获取负责人和预计时间；识别了至少一个"未知的未知"（应该知道但不知道的信息域） |

## S5 | 输出格式

```json
{
  "known_information": [
    {
      "item": "string",
      "category": "product | phenomenon | time | quantity | personnel | test_result | customer_feedback",
      "reliability": "confirmed | inferred | unverified",
      "source": "string",
      "key_for_investigation": false
    }
  ],
  "missing_information": [
    {
      "item": "string",
      "priority": "blocking | important | nice_to_have",
      "needed_for_stage": "D2 | D3 | D4",
      "acquisition_method": "string",
      "acquisition_owner": "string | null",
      "estimated_time": "string | null",
      "fallback_plan": "string | null"
    }
  ],
  "unknown_unknowns": [
    "string — domains of information we should know but don't"
  ],
  "confidence": 0.0
}
```

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 把推断和假设放进"已知信息" | 已知信息必须是直接观察/测量/记录的。推断和假设另列 |
| 缺失信息写空泛表述 | "需要更多数据"→ "需要 2404 批次的最终测试站电压测量原始数据" |
| 不区分阻塞性和锦上添花 | 阻塞性信息必须明确获取方案、负责人和时间承诺 |
| 假设所有信息同等可靠 | 每条已知信息标注可靠度：确认/推测/待验证 |
| 忽略"不知道什么"的诚实盘点 | 承认信息缺口不是弱点，忽视缺口才是 |
