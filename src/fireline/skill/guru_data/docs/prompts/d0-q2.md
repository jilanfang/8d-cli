# d0-q2 — 发现来源

## 行业背景

问题的发现方式决定了信息链的完整性和可信度。ASQ 8D 方法论和 VDA 第 4 卷均指出：发现来源需要记录到"具体个人 + 具体场景 + 具体渠道"的粒度，因为信息在传递中每经过一手就可能损失或失真。IATF 16949 第 7.5.3.2.1 条要求记录可追溯的工程信息。如果发现来源是"客户说产品有问题"但不知道客户方是谁、在什么工况下发现、通过什么渠道传过来的，D2 的问题定义很可能建立在二手甚至三手失真信息上。弱的发现溯源还会导致 D4 根因分析时找不到正确的复现条件。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **人员溯源**："第一个发现异常的人是谁？是客户方的操作员、质检员、采购？还是我们自己产线的作业员、IPQC？还是售后服务的工程师？有没有联系上这个人？"
2. **场景还原**："他在做什么操作的时候发现的？是产线正常生产中发现的，还是客户装机时发现的，还是终端用户使用中发现的，还是来料检验发现的？"
3. **渠道映射**："这个消息经过了几道传递？客户→客户采购→我们销售→我们质量→你？每一道分别是谁？你能不能画一条信息传递链出来？"
4. **时间精度**："发现的具体时间是什么？精确到小时最好。如果没有精确时间，至少精确到班次。这个时间能不能和 MES、ERP、邮件时间戳交叉验证？"
5. **一手信息获取**："你能不能直接和第一个发现异常的人通话或见面？如果不能，障碍是什么？"

## S2 | 第一性原理收敛

追问链 —— 从表面信息逐层深入：

**Layer 1 — 身份确认**
- "你说'客户发现的'—— 客户公司的谁？叫什么名字？什么岗位？有没有联系方式？"
- "你说'产线发现的'—— 哪条产线？哪个工位？哪个作业员？什么班次？"

**Layer 2 — 场景具体化**
- "'装机时发现'—— 装到哪个设备上？装到第几步发现的？正常装机步骤是什么？"
- "'检验时发现'—— 用的什么检验标准？什么量具？检验频率是多少？这个缺陷以前在这个检验点发现过吗？"

**Layer 3 — 信息衰减评估**
- "从第一个发现者到你这里，经历了几个信息中转站？每个中转站有没有添加自己的判断和解读？"
- "你现在能不能拿到第一发现者的原始记录或原话？如果拿不到，离原始信息最近的人是谁？"

## S3 | 反事实挑战

1. "如果第一个发现者看到的不是他认为的那个问题，而是另一个问题（比如他把安装不当导致的故障当成了产品缺陷），什么证据能在不回现场的情况下区分？"
2. "假设信息传递链中有一环误读了上一环的信息，最可能在哪一环出错？你怎么验证传递没有失真？"

## S4 | 答案评分

评分维度：**可追溯性**、**一手性**、**时间精度**

| 等级 | 标准 |
|------|------|
| Insufficient | 发现者不明确（"客户""产线"不是人名）；场景模糊；不知道发现时间；信息链不清晰 |
| Basic | 发现者姓名和岗位明确；场景可描述；时间精确到天或班次；信息传递路径清楚 |
| Excellent | 发现者可直接联系；发现场景可复现；时间可交叉验证（MES/ERP/邮件）；信息链中每一环有姓名；已获取第一发现者原话或原始记录 |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 接受"客户"作为发现者 | 追问客户方的具体姓名和岗位 |
| 把信息传递链当作黑盒 | 逐环画出：谁→谁→谁，每环标注传递方式 |
| 接受"大概是上周"这种时间精度 | 追溯邮件/系统时间戳，最差不低于班次精度 |
| 跳过场景描述直接进入技术分析 | 场景信息影响 D2 问题边界和 D4 复现条件 |
| 把二手解读当作一手观察 | 标记信息层级：一手观察 / 二手转述 / 三手总结 |
