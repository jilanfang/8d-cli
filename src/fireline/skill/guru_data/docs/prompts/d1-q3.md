# d1-q3 — 技术角色

## 行业背景

技术角色是 8D 团队中负责根因技术分析的人。Ford 8D 手册和 AIAG CQI-20 要求技术角色必须覆盖至少产品设计、制造工艺、测试验证三个领域中的相关方向 —— 单一领域的技术视角几乎必然导致根因搜索偏差。ASQ 指出：技术角色不匹配是"找不对根因"的第二大原因（第一是问题定义不清晰）。一个好的技术角色不是最资深的人，而是对该失效模式所涉及的物理机制、设计假设、制造变异最熟悉的人。如果让一个有管理头衔但没有技术深度的人担任技术角色，D4 的 5-Why 和机制追问会全部停在表面。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **领域匹配**："在这个失效模式涉及的技术领域里，公司内谁最懂？（不是职位最高的，是真正懂这个产品/工艺/失效机制的人）他是不是已经在团队里了？"
2. **多领域覆盖**："这个技术角色懂的是哪个领域？设计？工艺？测试？材料？如果失效涉及多个领域（比如设计+工艺耦合），有没有第二技术角色补位？"
3. **第一性理解**："这个人能不能用第一性原理解释这个产品是怎么工作的？不是背规格书，而是从物理机制（力/热/电/化学）上描述系统？"
4. **历史经验**："他以前有没有分析过类似的失效模式？有没有参与过这个产品或同类产品的 8D？"
5. **时间可用性**："这个技术专家平时的主要工作是什么？能不能从日常工作中抽出连续的时间（不是碎片化的 15 分钟）来做技术分析？需要谁分担他的日常工作？"

## S2 | 第一性原理收敛

追问链 —— 从表面答案逐层深入：

**Layer 1 — 能力验证**
- "你说'工艺工程师负责技术分析'—— 他对这个具体产品的工艺了解有多深？是不是每天都在产线？还是管多个产品线只偶尔看这个产品？"
- "你选的人擅长的是制造工艺，但如果根因最终指向设计假设（比如应力分析有误），他有没有能力做设计评审？还是需要拉一个设计工程师进来？"

**Layer 2 — 边界扫描**
- "除了你选的这个人，公司里还有谁对这个产品/失效模式有深度理解？如果这个人说'我分析不出来'，备选是谁？"
- "有没有考虑过供应商那边的技术资源？如果根因可能涉及来料，供应商的技术窗口是谁？"

**Layer 3 — 独立性质疑**
- "这个技术角色有没有可能是问题相关方？（比如他设计的产品出了故障，现在让他自己分析自己的设计问题）。如果是，你们怎么确保分析的客观性？"
- "如果技术角色的分析结论和他的上级观点冲突（比如上级认为是供应商问题，技术分析认为是设计问题），他有没有独立发表结论的空间？"

## S3 | 反事实挑战

1. "假设你选的技术角色实际上对这个失效机制的理解不够深（他自己不知道，你也不知道），你在什么时候会发现这件事？通过什么信号？—— 比如追问到第三层物理机制时他回答不出来。"
2. "如果你选的这个人因为太熟悉这个产品，反而看不到'熟悉盲区'（比如某个长期存在但被习惯性忽略的设计问题），谁来当那个'不知道历史所以敢问蠢问题'的人？"

## S4 | 答案评分

评分维度：**领域匹配度**、**多领域覆盖**、**独立性**

| 等级 | 标准 |
|------|------|
| Insufficient | 技术角色未明确；选人理由是"他是经理"而非技术匹配；只覆盖单一领域；有明显利益冲突但未处理 |
| Basic | 技术角色具名；覆盖至少一个相关技术领域；了解该产品/工艺；有时间投入计划 |
| Excellent | 覆盖失效涉及的所有关键领域（设计/工艺/材料/测试）；有第一性原理解释产品工作原理的能力；有同类失效分析经验；有利益冲突防护机制；有备选技术资源池 |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 按职级选人而非技术匹配度 | 最懂失效机制的人可能是技术员而非经理 |
| 只安排一个领域的技术角色 | 如果失效可能涉及多个领域，明确第二技术角色和触发条件 |
| 让问题相关方担任唯一技术分析角色 | 如果技术角色恰好是问题根因可能指向的人，建立独立评审机制 |
| 选了技术专家但没确认他的时间 | 顶尖专家往往最忙。必须确认时间分配，必要时做工作分担安排 |
| 假设一个人覆盖所有领域 | 诚实列出领域覆盖缺口。缺口不是问题，不承认缺口才是 |
