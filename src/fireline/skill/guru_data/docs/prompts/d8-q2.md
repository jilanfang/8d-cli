# d8-q2 — 经验总结与 Truth Chain

## 行业背景

一个 8D 做完之后最有复用价值的产出不是报告本身，而是 Truth Chain——一条从现象到根因到措施到验证的完整因果链。AIAG CQI-20 强调：有效的 lessons learned 系统必须能让后续的类似问题"站在前人的肩膀上"，而不是每次都从零开始调查。IATF 16949 第 7.3.3.1 条要求组织将以往的设计/过程经验应用于新产品开发——但如果没有结构化的 Truth Chain，所谓的"经验"只是零散的回忆和口头传说。

8D 实践中，经验总结的典型失败模式有两种。一是"空话模式"：教训写成了"加强培训、提高意识、完善管理"，这些放在任何 8D 里都能用，等于什么都没说。二是"孤立模式"：经验只存在这份 8D 报告里，没有进入知识库、培训体系或设计规范，下一个项目启动时完全看不到。

D8 的元学习还有一个经常被忽略的层面：**这次 8D 过程本身有什么可以改进的？** 不只是问"产品出了什么问题"，还要问"我们做 8D 的方式出了什么问题"——哪一步做浅了、哪个关键证据来晚了、哪个讨论兜了圈子。

## S1 | 发散探索 — 多角度触发器

1. **Truth Chain 编码**：从现象到根因到措施到验证，形成一条完整、可独立阅读、可被未来工程师引用的因果链。每条节点都要有足够的上下文，让一个没参与这个 case 的人能看懂逻辑。

2. **做得好的部分**：这个 8D 过程中哪些做法是有效的、值得在未来类似调查中重复使用的？例如：某个分析工具用得好、某个跨部门协作机制跑通了、某个数据源帮了大忙。

3. **可以改进的部分**：哪个环节走了弯路？哪里的信息获取太慢？哪个决策在没有充分数据的情况下就做了？如果再来一次类似的调查，怎么做会更高效？

4. **可复用的知识**：从这个 case 中提取出来的知识，除了挂在本次 8D 报告里，还能用在哪些场景？新项目开发、供应商评审、内审检查表、工程师培训案例？

5. **不确定性声明**：这个 case 里我们还有什么没搞清楚的？哪些结论是"基于当前证据合理推断"但还没有被决定性验证的？明确声明不确定性比假装一切确定更有价值——下一个调查员看到不确定性声明，可以省掉重复的试错。

## S2 | 第一性原理收敛

- 这次 8D 中最有价值的洞察是什么？不是"问题被解决了"这个结果，而是"我们搞清楚了某种因果关系"这个认知增量。这个增量是否被写下来了？一个没参与调查的人能不能从 truth chain 中获得同样的认知增量？

- 这个 case 的 lessons learned 和之前其他 8D 的 lessons learned 有没有重复？如果连续三个 8D 的教训都是"变更评估不充分"，说明组织层面没有从个案中学到东西——这时应该触发流程级改进，而不是再记一遍同样的教训。

- 如果这个 truth chain 被放进知识库，未来的工程师在什么场景下能检索到它？靠什么关键词、什么分类、什么触发条件？如果检索不到，等于没有沉淀。

## S3 | 反事实挑战

- 把 lessons learned 里的每一条拿出来，问：如果下一份 8D 报告直接复制粘贴这一条，它还有意义吗？如果答案是"有"，说明这条太泛了，必须重写。
- 把 truth chain 给一个没参与这个 case 的同事看，10 分钟后问他：问题的因果链是什么？最关键的那一步是什么？如果他答不上来，说明 truth chain 写的不够清楚。
- 如果这个 case 的经验教训和公司内部已有的某个设计规范/工艺标准冲突（例如 case 证明某个参数必须收紧，但规范里还是旧值），谁会负责推动规范更新？如果不指定这个人，规范永远不动。
- 假设 3 年后这个产品线的人全换了，新团队还能从这个 8D 里学到东西吗？还是"前人的 8D 报告存在服务器上，但从来没人看"？

## S4 | 答案评分

| 维度 | Insufficient | Basic | Excellent |
|------|-------------|-------|-----------|
| **Truth Chain 完整性** | 只有现象和措施，中间没有因果链 | 现象→根因→Escape Point→措施→验证 五段齐全 | 五段齐全 + 每段有足够的上下文和证据引用 |
| **教训具体性** | "加强培训/提高意识/完善管理" | 具体说明了从什么改到什么、学到了什么 | 具体教训 + 可检索标签 + 触发场景（什么情况下应引用这条教训） |
| **元学习** | 没有反思 8D 过程本身 | 指出了 1 个过程改进点 | 2+ 过程改进点 + 具体建议（下次怎么做） |
| **不确定性声明** | 未提及任何不确定性 | 列出了未解决/未验证的项 | 每项不确定性 + 为什么不确定 + 什么条件下可以解决 |

不达 Basic 不能进入 S5。lessons_learned 中如出现"加强培训"、"提高意识"、"完善管理"、"举一反三"等空泛表述，直接 Insufficient。

## 纠偏原则 / 禁止项

- **禁止**：lessons_learned 中出现"加强培训"、"提高意识"、"完善管理"、"举一反三"、"吸取教训"、"重视质量"等空话。每条教训必须具体到"从什么事中学到了什么具体认知"。
- **禁止**：truth_chain 跳过中间步骤。不能现象直接跳到措施——必须经过根因和 Escape Point。
- **禁止**：把"问题解决了"当成元学习。元学习问的是"我们做 8D 的方式哪里可以更好"，不是"问题解决了没"。
- **禁止**：没有不确定性声明。每个 8D 都有未解决的问题或未被决定性验证的结论——诚实记录比假装完美更有工程价值。
- Truth Chain 是这个 case 最有复用价值的产出。如果这项写不好，整个 8D 的知识沉淀价值归零。
