# d7-q3 — 横向展开与组织学习

## 行业背景

Yokoten 的第二层是"有效措施复制"——不仅检查同类产品有没有风险（d7-q1），还要把已经验证有效的纠正措施主动部署到相似环境。IATF 16949:2016 第 7.3.3.1 条（产品设计输入）和第 10.2.3 条均隐含了 lessons learned 必须跨产品/跨项目应用的要求。AIAG CQI-20（Effective Problem Solving）明确指出：组织学习的质量决定了同样问题会不会换个产品编号再来一遍。

横向展开失败的最大根源不是技术问题，而是组织问题——措施在 A 产品上跑通了，但 B 产品的 Owner 不知道、没动力、或者认为"我们是特殊的"。真正的横向展开需要打破产品线之间的信息孤岛。

另一个经常遗漏的层面是**组织学习**：在这个案例中，是什么管理决策、假设或压力让这个漏洞在系统里存在了这么久？如果不回答这个问题，措施只打在了技术层，组织层的漏洞依然开着。

## S1 | 发散探索 — 多角度触发器

1. **产品/产线层面**：哪些产品/产线/工厂与本次问题点共享了同一个根因的前提条件？不是因为结构相似（那是 q1 的事），而是因为同一个管理逻辑、同一套工艺规范、同一个变更流程在起作用。

2. **组织决策层面**：这个问题的根因能追溯到某个管理决策或资源分配吗？例如：为了降本把检验频次砍了？为了赶交付跳过了某个 review？这个决策在其他项目/产品线上也生效吗？

3. **流程假设层面**：当前流程设计中有一个假设后来被证明不成立——这个假设是什么？它只在当前产品上被推翻，还是在其他产品上同样存在？

4. **能力复制层面**：这次有效的纠正措施（D5 中已实施并 D6 验证通过的）能否直接复制到其他产品线？复制成本、技术障碍、组织阻力分别是什么？

5. **知识沉淀层面**：除了更新文件（q2），这次的经验有没有进入培训体系、工程标准、设计规范、供应商准入标准？还是仅限这个 8D 报告内部？

## S2 | 第一性原理收敛

追问组织/系统层面的根因：

- 为什么同类产品/产线之前没有自动受益于 A 产品上的教训？是缺少横向信息传递机制，还是有人传了但被"我们不适用"挡回去了？
- 当前组织里，谁有权限和动机推动跨产品线的措施复制？如果这个人不存在或不作为，机制上怎么补？
- 这个漏洞在系统里存活了多久？从漏洞产生到今天被 8D 关闭，中间有多少次机会可以更早发现？为什么错过了？
- 管理层的什么优先序决策（成本/交付/质量 的权衡）间接创造了这个漏洞的生存空间？这个权衡在被这次教训修正了吗？

## S3 | 反事实挑战

- 如果横向展开只靠发邮件——"请大家参考本次 8D 自查"——实际执行率会是多少？过去的类似邮件，有多少人真的打开并行动了？
- 假设 B 产品的 Owner 说"我们不一样，不需要这个措施"，你用什么证据判断这句话是真合理还是在抵抗改变？
- 如果这个 8D 的 truth chain 明年被调出来用于一个新项目，新项目的工程师能在 10 分钟内理解发生了什么，并判断自己的项目是否适用吗？
- 如果这次不把组织学习的结论写进管理层评审纪要，下一次战略会上这个教训还会被提及吗？如果不会，意味着什么？

## S4 | 答案评分

| 维度 | Insufficient | Basic | Excellent |
|------|-------------|-------|-----------|
| **展开具体性** | "展开到所有相关产品"——没有清单 | 列出了具体的目标产品/产线/工厂 | 清单 + 适用性评估（为什么选这些，为什么不选那些） |
| **措施可复制性** | 直接把 D5 措施名抄一遍，没有评估适配性 | 说明了措施是否需要适配、适配什么 | 适配方案 + 技术可行性 + 组织可行性评估 |
| **组织学习深度** | 没有提管理/决策层原因 | 识别了一个管理层因素但停在表面 | 具体的决策/假设/压力 + 修正该因素的行动 |
| **知识沉淀** | "已记录 lessons learned"——没有说明记录在哪、谁可以用 | 说明了知识存放位置和检索方式 | 知识进入标准/规范/培训/IT 系统的具体路径 + 可检索性验证 |

不达 Basic 不能进入 S5。必须有具体的目标产品/产线列表，不能只有"全部展开"。

## 纠偏原则 / 禁止项

- **禁止**："展开到所有产品/所有产线"——没有清单的横向展开是空话。
- **禁止**：只有展开目标，没有责任人和 deadline。没有 owner 的部署计划不会自己执行。
- **禁止**：跳过组织学习。如果你的根因追到了管理决策层，横向展开也必须打到管理决策层。
- **禁止**：把"发了邮件通知"当成横向展开完成了。必须验证目标产品/产线实际实施了措施。
- 如果一个目标被排除，必须写清楚排除逻辑。不能因为"太麻烦"或"他们不归我管"就把风险留给别人。
