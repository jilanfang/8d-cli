# d2-q1 — 客户视角问题描述

## 行业背景

D2 问题定义的第一步是"用客户的语言描述问题"。Ford 8D 手册和 AIAG CQI-20 都明确要求：客户视角和内部视角必须分开陈述，且客户视角优先。原因很简单：如果客户说他收到的是"不亮的产品"，而内部定义为"输出电压 0V"，这两个描述指向的可能是完全不同的问题（前者可能是电源问题，后者可能是测量点问题）。IATF 16949 要求组织准确记录客户投诉内容。客户视角的问题描述是 IS/IS-NOT 矩阵的输入，也是最终验证"问题是否解决了"的基准 —— 客户不认你的工程定义，只认他的使用体验。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **客户原话**："客户在投诉、邮件、电话里具体怎么说的？能不能逐字引用？不要帮我总结，给我原文。哪怕原文'不专业'、'情绪化'，那才是真正的客户声音。"
2. **客户操作链**："客户在用这个产品做什么？在什么操作步骤中发现的？正常操作应该是什么结果，实际发生了什么？这个操作在客户的业务流程里有多重要？"
3. **客户直观影响**："这个问题给客户造成了什么直接的后果？停线了？终端用户投诉他了？他的交付延迟了？他多花了人工去分拣？"
4. **客户复现描述**："客户说这个问题是每次都出现还是偶尔出现？在什么条件下出现？他自己有没有试过什么排查步骤？"
5. **客户期望**："客户明确说过他期望什么样的结果？他要求的围堵时间是多长？他认为什么是'解决了'？（换货？赔偿？永久预防？）"

## S2 | 第一性原理收敛

追问链 —— 从表面描述逐层深入：

**Layer 1 — 语言还原**
- "你刚才说的客户描述，是客户的原话还是你翻译过的？如果是翻译过的，客户原话用的是什么语言？能不能直接把原文（中文/英文/其他）贴过来？"
- "客户有没有发过照片或视频？这些影像里客户标注了什么位置？什么异常？"

**Layer 2 — 场景具体化**
- "客户说的'产品'具体是指哪个料号？他有没有提供产品标签的照片？如果他说的型号和我们的料号对不上，你怎么确认说的是同一个产品？"
- "客户是在什么环境/工况下使用这个产品的？这个工况和我们的设计工况一致吗？如果客户在超规格使用，他知道吗？"

**Layer 3 — 客户情绪与业务影响**
- "客户现在的情绪状态是什么？'不满但不急'还是'非常愤怒要求立刻解决'？这个判断是听谁说的，还是你直接跟客户聊过？"
- "从客户的业务角度看，这个问题影响了他的什么指标？（产线 OEE？终端用户 NPS？交付准时率？质量 PPM？）"

## S3 | 反事实挑战

1. "假设客户描述的问题其实不是产品问题，而是客户的使用方法不对（比如装反了、用错电压了、没按说明操作）—— 你现有信息里有什么线索支持或排除这种可能？"
2. "如果客户把他看到的现象描述给不同的人听，每个人转述出来会不会不一样？你最担心哪一层转述会加进自己的解读？"

## S4 | 答案评分

评分维度：**原声保真度**、**场景完整性**、**客户影响清晰度**

| 等级 | 标准 |
|------|------|
| Insufficient | 没有客户原话，全是内部总结；不清楚客户的使用场景；不知道客户的业务影响 |
| Basic | 引用了客户原话或原文；描述了使用场景和操作链；客户影响已说明（停线/分拣/投诉） |
| Excellent | 客户原话逐字引用（附原始邮件/记录）；客户使用场景与设计工况的差异已对比；客户情绪和业务指标影响已了解；客户明确期望已记录 |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 用内部工程语言替换客户原话 | 两套都要：客户原话 + 工程翻译。两者并存 |
| 在客户场景不明确时开始定义问题 | 先搞清楚客户在什么条件下用的、怎么发现的 |
| 忽略客户情绪和业务影响 | 8D 不只是解决技术问题，也是恢复客户信任 |
| 假设客户用法和设计工况一致 | 主动对比：客户实际工况 vs 设计规格 |
| 总结客户描述而非引用 | "客户说电源有问题" vs "客户邮件原文：The unit does not power on..." |
