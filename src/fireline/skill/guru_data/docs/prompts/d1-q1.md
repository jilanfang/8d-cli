# d1-q1 — Owner

## 行业背景

8D 方法论对 Owner 有严格定义：Owner 是对 8D 过程和最终结果的单一问责人（Accountable），不是"负责协调的部门"。ASQ 和 Ford 8D 手册均要求 Owner 必须是具名个人，且有权调动跨职能资源。RACI 模型中，Owner 是那个承担 "A" 角色的人 —— 最终签字确认根因、措施和结案的人。AIAG CQI-20 特别指出：Owner 指定不明确是 8D 超期或方向漂移的首要组织原因。如果 D1 阶段 Owner 是"质量部"而非具体个人，审核员会直接开出不符合项。一个弱的 Owner（无权调动资源、无法拒绝不合理的催促、不敢推翻上级先入为主的判断）会让整个 8D 失去技术严肃性。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **责任边界**："谁对这个问题的最终调查结论和纠正措施效果负责？不是谁'参与'或'协助'，而是如果 8D 没做好，谁被问责？"
2. **组织权力**："这个 Owner 有没有权力从生产、质量、技术、供应商管理这几个职能调人进团队？有没有权力拒绝管理层不合理的结案时间压力？"
3. **时间承诺**："Owner 能在这件事上投入多少时间？是全职投入还是兼职？如果是兼职，哪些日常任务会抢走他的注意力？需要谁分担他的日常工作？"
4. **过往经验**："Owner 以前有没有主导过 8D？有几个完整的案例？从启动到结案的平均周期是多长？最大的经验教训是什么？"
5. **替代方案**："如果这个人中途离职、生病或被调走，谁是最合适的替代者？这个人之间有没有交接过？"

## S2 | 第一性原理收敛

追问链 —— 从表面答案逐层深入：

**Layer 1 — 身份确认**
- "你说'质量经理负责'—— 具体是哪位质量经理？全名是什么？他确认自己同意担任 Owner 吗？"
- "你说'我来当 Owner'—— 你有没有直接向上级确认过你的时间和授权范围？上级知不知道这个 8D 需要你多少时间？"

**Layer 2 — 权力验证**
- "如果 D4 的根因指向另一个部门（比如研发的设计问题或采购的供应商选择问题），Owner 有没有权力要求那个部门提供数据和配合验证？"
- "如果 Owner 认为需要更多时间才能找到根因，但客户要求 30 天结案，Owner 有没有权力和管理层/客户谈判延期？还是会被迫提前结案？"

**Layer 3 — 资源缺口**
- "Owner 有没有预算权？（外部实验室送检、第三方分析、出差去现场）如果没有，谁审批？审批周期多长？"
- "除了 Owner 自己，他能调用的核心成员目前各自承担多少日常工作？有没有可能被抽走？"

## S3 | 反事实挑战

1. "假设 Owner 在 D4 阶段发现根因指向他所在的部门自己（比如质量检验漏检导致了流出），他有没有可能下意识地回避这个方向？你们怎么防止这种自我审查？"
2. "如果 Owner 认为自己被指定为 Owner 是不合适的（比如他不懂这个产品的技术细节），他有没有安全地说'不'的渠道？现在就说'不'比 D4 才发现不懂要好。"

## S4 | 答案评分

评分维度：**具名性**、**授权充分性**、**独立性**

| 等级 | 标准 |
|------|------|
| Insufficient | Owner 是部门名而非个人名；Owner 不清楚自己需要做什么；无时间承诺；无权调动跨职能资源 |
| Basic | Owner 是具名个人；已确认同意担任；了解 8D 流程要求；可以调动至少生产/质量/技术三个职能的资源 |
| Excellent | Owner 有书面授权（至少邮件确认）；有时间分配计划（明确日常任务的替代安排）；有替代 Owner 预案；Owner 所在的部门不是问题发生部门（或已建立利益冲突防护机制） |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| Owner 写"质量部""技术部" | 必须是具体姓名。部门可以写，但人必须是具体的 |
| 默认让质量经理当 Owner | 根据问题性质选择 Owner：设计问题→技术 Owner、供应商问题→SQE Owner |
| Owner 恰好是问题发生部门的人 — 不做利益冲突评估 | 如果 Owner 部门 = 问题根因可能指向的部门，建立独立评审机制 |
| 接受"我们团队一起负责" | 8D 必须单一问责人。团队协作 ≠ 团队负责制 |
| Owner 不懂 8D 流程也让他当 | 确认 Owner 了解 8D 流程，必要时先培训或配 8D 教练 |
