# d2-q6 — 事实与推断分离

## 行业背景

这是整个 D2 阶段最重要、也是 8D 实践中被违反最多的纪律。ASQ 8D 方法论和 Kepner-Tregoe 问题解决框架都严格区分三个层级：Fact（可直接观察或测量的事实）、Inference（基于事实的逻辑推论）、Assumption（未经证实的假设）。IATF 16949 审核中，把推断当成事实来定义问题是最致命的错误类型 —— 因为一旦推断被当成事实写进 D2 问题定义，D4 根因分析的搜索空间就被这个未经验证的推断框定了，真正的根因可能永远在搜索范围之外。Ford TOPS 方法论特别强调：如果 D2 阶段不做好事实/推断分离，8D 会变成"证明自己一开始就猜对了"的自我论证循环。

典型错误场景：团队在 D2 写了"根本原因是焊接工艺参数偏移"，这其实连推断都不是，是跳到 D4 的结论。正确的 D2 事实是"X 射线显示 5 个不良品的焊点空洞率 > 25%（规格 ≤ 15%）"，推断是"空洞率过高可能导致焊点疲劳寿命下降"，假设是"参数偏移导致了空洞率升高"。

## S1 | 发散探索 — 多角度触发器

Agent 从以下问题中选 2-3 个向用户提问。

1. **逐条审查**："把你刚才说的关于这个问题的所有陈述，一句一句读出来。每条前面标注：这是 Fact（你能测量/观察到），还是 Inference（你基于某些事实推断的），还是 Assumption（你假设的、还没验证过）。"
2. **证据溯源**："对每一个 Fact，它的证据是什么？谁观察的/测量的？用什么工具？什么时候？在什么条件下？如果这条 Fact 被质疑，你能不能立刻拿出证据？"
3. **推断链追踪**："对每一个 Inference，它是基于什么 Fact 推断出来的？推断逻辑是什么？（比如'因为焊接温度偏低，所以推断焊点强度不够'—— 逻辑链条是什么？有没有跳过步骤？）"
4. **假设暴露**："有没有一些事情你们团队默认是'对的'或'知道的'，但实际上没有人验证过？比如'这个设计没问题，因为用了三年了''这个供应商一直很稳定'—— 这些都是 Assumption。"
5. **推断转化为验证计划**："你列出的每一个 Inference 和 Assumption，有没有对应的验证计划？在什么时候、用什么方法、谁来验证？没有验证计划的推断不能进入下一个 D。"

## S2 | 第一性原理收敛

追问链 —— 从表面分离逐层深入：

**Layer 1 — 分离训练**
- "你说'不良品是焊接问题'—— 这是事实还是推断？事实是'X 光发现 5 个样品焊点有超过 25% 的空洞'。'焊接问题'是对这些空洞的来源推断。你的事实是什么？"
- "你说'操作员失误导致装反'—— 操作员做了什么具体动作你看到了？还是你看到产品装反了，推断是操作员做的？这两个是完全不同的事。"

**Layer 2 — 隐藏推断挖掘**
- "最危险的不是明显的推断，而是那些你根本没意识到是推断的陈述。比如'产品设计没问题'—— 这是陈述还是假设？你验证过设计吗？"
- "有没有一些推断是你从别人的推断里推出来的？（二手推断 > 一手推断 > 事实）。这个推断链越长，可信度越低。"

**Layer 3 — 推断的后果分析**
- "如果你列出的某个关键 Inference 实际上是错的，你的整个问题定义会受影响吗？受影响程度多大？（局部修正 vs 全部推翻）"
- "哪个 Inference 在你们团队里有不同意见？有人觉得是 A 原因，有人觉得是 B 原因 —— 这说明两个都是 Inference，不是 Fact。"

## S3 | 反事实挑战

1. "选出你列表里最重要的一条 Inference，假设它是错的。如果它是错的，你最快能用什么方法证明？（设计一个可以推翻它的实验或检查）"
2. "假设一个外部审核员拿到你的 Fact 列表，他会不会说'你列出来的所谓 Fact 之一，其实也是 Inference'？你觉得哪一条最容易被挑战？为什么？"

## S4 | 答案评分

评分维度：**分离彻底性**、**推断溯源**、**验证计划**

| 等级 | 标准 |
|------|------|
| Insufficient | 事实和推断混在一起；"应该是""肯定是"等判断被当成事实；推断没有基于具体事实；假设未标注 |
| Basic | 事实和推断已分离并列在两个列表中；每个推断标注了所基于的事实和推理逻辑；未经验证的假设已识别并标记；核心推断有验证计划 |
| Excellent | 每条事实有证据溯源（谁/什么方法/什么时间/什么条件）；推断链不超过两步（事实→推断→推断不行）；团队内不同推断已列出并对比；每条关键推断和假设都有反事实验证计划；识别并标出了"隐藏推断"（团队默认但未验证的信念） |

## 纠偏原则 / 禁止项

| 禁止 | 替代方向 |
|------|---------|
| 把"我觉得""应该是""肯定是"当事实 | 诚实回答：你观察/测量到了什么？不只是你认为什么 |
| 推断不标注基于什么事实 | 每个推断必须链接到至少一个具体事实 |
| 推断链过长不检查 | 二手推断（基于推断的推断）要单独标出，可信度显著降低 |
| 团队默认的信念不暴露 | 主动挖掘"大家都觉得没问题所以没查过"的假设 |
| 推断没有验证计划 | 所有推断必须标记验证状态。未验证的推断不能进入 D4 作为根因候选的基础 |
| "这个问题很明确" 代替了分离过程 | "明确"往往意味着大量推断被当成了事实。越是觉得"明确"，越要做事实/推断分离 |
