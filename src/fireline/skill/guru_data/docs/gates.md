# Gate 检查清单 v0.2.1

Agent 必须在进入下一阶段前，逐项确认当前阶段的 gate 条件。任一条件不满足，都必须停止并返回 blocker。

## 评分阻断规则（适用所有 D）

- 任何必答题被评 **Insufficient** → 不能进入 S5，不能推进阶段。
- 任何必答题被评 **Basic** → 可以推进，但 Agent 应提示达到 Excellent 还差什么。
- S3 反事实挑战未执行 → 不能进入 S4。

## D0 → D1

- 已确认客户报告的现象（d0-q1），且评分 ≥ Basic。
- 已确认发现来源（d0-q2），且评分 ≥ Basic。
- 已评估严重度/是否启动完整 8D（d0-q3），且评分 ≥ Basic。
- 已记录紧急处置措施（d0-q4，如无则标记 N/A）。

## D1 → D2

必须全部满足：

- [ ] d1-q1 Owner 已指定具体个人（非部门名），评分 ≥ Basic。
- [ ] d1-q2 客户接口人已指定具体个人，评分 ≥ Basic。
- [ ] d1-q3 技术角色已指定具体个人或岗位，评分 ≥ Basic。
- [ ] 团队最少覆盖 production + quality + engineering 三个职能。

不满足时提示：
- "D1 缺少 Owner，不能进入 D2。"
- "D1 缺少客户接口人，不能进入 D2。"
- "D1 缺少技术角色，不能进入 D2。"
- "D1 团队缺少 [生产/质量/工程] 职能视角，不能进入 D2。"

## D2 → D3

必须全部满足：

- [ ] d2-q1 客户视角的问题描述已完整，评分 ≥ Basic。
- [ ] d2-q2 厂内视角的问题描述已完整（或标记为待补充并说明原因）。
- [ ] d2-q4 影响范围已量化（数量、批次、金额、客户影响），评分 ≥ Basic。
- [ ] d2-q6 事实与推断已分离，且至少有一个可观察/可测量的事实，评分 ≥ Basic。
- [ ] Is/Is-Not 矩阵至少覆盖 3 个维度（What/Where/When/Who/HowMany）。

不满足时提示：
- "D2 问题定义不完整：缺少 {具体 question}。"
- "D2 只有推断没有事实，不能进入 D3。"
- "D2 事实/推断/假设未分离，不能进入 D3。"

## D3 → D4

必须全部满足：

- [ ] D1 的必填项完整（Owner、客户接口、技术角色）。
- [ ] D2 的问题定义完整。
- [ ] d3-q1 ~ d3-q5 中至少识别出 2 个风险窗口（客户现场、在途、库存、WIP、上游），每个涉及窗口的评分 ≥ Basic。
- [ ] Quality Alert 状态已确认（已发出或不需要）。

不满足时提示：
- "D3 风险窗口识别不足，至少需识别客户现场/在途/库存/WIP/上游中的 2 项。"

## D4 → D5

必须全部满足：

- [ ] d4-q2 候选原因已列出，每个高优先级原因有证据支持和反驳，评分 ≥ Basic。
- [ ] d4-q3 5-Why 已追到控制点失守（≥ 3 层），根因不停在"人"/"供应商"/"培训"，评分 ≥ Basic。
- [ ] d4-q4 Escape Point 已识别，找到最早应拦住问题但未拦住的控制点，评分 ≥ Basic。
- [ ] d4-q6 控制点判断已输出 + MRC（管理系统/业务流程/人员/组织）至少覆盖一个维度，评分 ≥ Basic。
- [ ] S3 反事实挑战已在 d4-q3 和 d4-q4 上执行。

不满足时提示：
- "D4 缺少候选原因。"
- "D4 缺少发生原因（5-Why），或 5-Why 停在责任归属未追到控制点。"
- "D4 缺少流出原因（Escape Point）。"
- "D4 缺少控制点判断或 MRC。"
- "D4 根因未经过反事实挑战验证，不能进入 D5。"

## D5 → D6

必须全部满足：

- [ ] 每项纠正措施都对应一个已确认的根因（TRC/MRC/Escape Point），d5-q1/d5-q2 评分 ≥ Basic。
- [ ] 所有措施都具体可执行，不含"加强培训/检验/管理"等空泛表述，d5-q3 评分 ≥ Basic。
- [ ] 每项措施已标注 Poka-Yoke 等级（消除/检测/减轻）。
- [ ] 已评估每项措施的技术可行性和成本可行性，d5-q4 评分 ≥ Basic。
- [ ] 每项措施都有明确责任人和完成时间，d5-q5 评分 ≥ Basic。
- [ ] D5 Verification 计划已定义（试验条件），区别于 D6 Validation。

不满足时提示：
- "D5 措施未对应到根因/Escape Point。"
- "D5 措施不够具体：[具体措施名称]——缺少做什么/谁做/何时完成/可验证标准。"
- "D5 措施未做 Poka-Yoke 等级评估。"
- "D5 缺少可行性评估。"
- "D5 缺少责任人或完成时间。"

## D6 → D7

必须全部满足：

- [ ] d6-q1 验证方法已具体说明（方法名称+测量指标+MSA状态），评分 ≥ Basic。
- [ ] d6-q2 验证范围已明确（样本量需有统计依据/覆盖D2全部受影响范围），评分 ≥ Basic。
- [ ] d6-q3 判定标准已提前定义（可量化，含通过/失败/不确定三级），评分 ≥ Basic。
- [ ] d6-q4 验证结果有实际数据支撑，不能只有"已验证有效"，评分 ≥ Basic。
- [ ] 实际执行与计划的一致性已审查，偏差已解释。

不满足时提示：
- "D6 缺少验证方法。"
- "D6 缺少验证范围或样本量无统计依据。"
- "D6 缺少判定标准或标准不可量化。"
- "D6 验证结果没有数据支撑。"

## D7 → D8

建议满足：

- [ ] d7-q1 已检查同类产品/工序/供应商是否存在相同风险，有具体检查结果。
- [ ] d7-q2 已列出需要更新的系统文件（FMEA/控制计划/SOP/检验标准），PFMEA和控制计划已同步。
- [ ] d7-q3 已规划横向展开，有具体目标列表和责任人或明确排除理由。

## closed（结案）

必须全部满足：

- [ ] D4 完整（TRC + MRC + Escape Point + 控制点），所有评分 ≥ Basic。
- [ ] D5 所有措施评分 ≥ Basic，Poka-Yoke 等级已标注。
- [ ] D6 验证结果完整且有数据，评分 ≥ Basic。
- [ ] D7 横向展开已完成或已规划，系统文件更新已启动。
- [ ] D8 结案条件逐项确认，Truth Chain 已编码，不确定性已声明。
- [ ] Owner + 技术角色 + 客户接口均已签署。

不满足时提示：
- "D6 缺少验证数据，不能结案。"
- "D4 根因分析不完整，不能结案。"
- "D8 结案条件未满足：[具体缺失项]。"
- "必签角色未完成签署：[角色名]。"

## Profile Gate 聚合规则

**当行业 profile 已加载时，Agent 必须在每个 D→D+1 转换时合并两类 gate 条件：**

1. **基础 gate**（本文）：方法论层面的阻断条件（评分不足、探索方向数不达标、5-Why 深度等）
2. **行业 gate**（profile overlay 的 Hard Gates 段）：条款层面的阻断条件（PPAP 影响评估、特殊特性控制、OEM 时限等）

**聚合方式：**
- 两类 gate 的逻辑是 AND——两者都通过才能进入下一 D
- 如果基础 gate 通过但行业 gate BLOCK → 不能推进，阻断信息来自行业 gate
- 如果加载了多个 profile（如 ISO 9001 + IATF 16949）→ IATF 的更严格条件覆盖 ISO 9001 的同领域条件（IATF 是 ISO 9001 的补充）
- 在 gate 状态输出中，分别标注 base_gates 和 profile_gates 的通过/阻断状态

**Profile Hard Gates 位置：**
- ISO 9001:2015 → `profiles/iso9001/overlay.md` → Hard Gates 段
- IATF 16949 → `profiles/iatf16949/overlay.md` → Hard Gates 段
