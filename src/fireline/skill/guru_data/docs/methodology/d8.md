# D8 — 结案

## 行业最佳实践

D8 在行业里有两个常见极端：一端是"签个字就结了"，另一端是"永远结不了因为报告不完美"。好的 D8 在这两者之间 — 承认不确定性、承认限制、但不回避结论。

行业标准要求五项结案动作：

1. **团队自评**：按 Insufficient / Basic / Excellent 三级自评每个 8D 步骤的质量。这不是管理层的评语，而是团队自己的判断。自评的不一致（有人打 Insufficient 有人打 Excellent）本身就是信号，说明团队对质量标准认知不统一。
2. **Truth Chain 编码**：把整个 8D 的因果链编码为一条可追溯的线索：现象 -> 根因 -> 流出点 -> 纠正措施 -> 验证证据 -> 经验。这条链是 8D 的"基因序列"，未来任何人读这条链都应该在 5 分钟内理解案例全貌。
3. **不确定性声明**：诚实列出"我们现在还不确定什么"。例如：验证样本量偏小、边界条件没覆盖到、长期可靠性数据不够、供应商端的根因依赖对方自查。这不是缺陷，是透明度。
4. **元学习**：跳出具体案例，反思"我们做 8D 的过程本身有什么可以改进的"。例如：D2 拖太久因为数据分散在三个系统里、D4 走了弯路因为一开始就跳到了结论、D7 横向检查没有明确标准导致来回讨论。这比"吸取教训"高一层 — 是"关于如何吸取教训的教训"。
5. **管理层签核**：不是橡皮图章。签核人确认三件事：报告逻辑链完整（Truth Chain 无断点）、团队自评诚实、剩余不确定性已被识别且有跟踪计划。签核不代表问题永不再发生，只代表"以当前所知，该做的都做了"。

## 探索方向矩阵

|探索方向|示例问题|
|------|------|
|结案条件核验|D4 根因是否完整（TRC + MRC + Escape Point 都有）？D5 措施是否每条对应根因？D6 验证数据是否满足标准？D7 横展是否落地？哪个条件最弱？|
|团队自评质量|自评的打分有没有出现"全 Excellent"？如果有人打 Insufficient，那个步骤的差距是什么、谁来跟、什么时候再评？|
|Truth Chain 完整性|从现象到经验这条链，哪一段最薄？缺哪一环就接不上了？如果 3 个月后新同事只读这条链，能不能还原关键决策？|
|不确定性管理|现在不确定的事情里，哪些是"进一步验证就能消除的"、哪些是"本质上不可消除的"？不可消除的不确定性怎么管理 — 靠定期复检还是靠设计余量？|
|元学习捕获|这次做 8D 的过程中哪个步骤最费劲？是缺数据、缺方法、还是缺人？这个费劲的点下次能不能通过流程改进来消除？|
|归档与可追溯|完整报告和附件存在哪里、用什么文件名编号、谁能在什么权限下访问？3 年后需要引用这个案例时，怎么在 5 分钟内找到？|
|复检计划|这个问题多久后需要复检一次？复检看什么指标、谁来做？如果复检发现问题，是开新 8D 还是回溯旧 8D？|

## 第一性原理锚点

D8 的本质问题是：**我们有没有在承认不确定性的前提下，把这次痛苦的完整因果链存进组织的长期记忆？**

追问到这个层才算到位：

- 如果 6 个月后这个问题换了形式再次出现，新接手的人能不能在 30 分钟内通过 Truth Chain 和 Lessons Learned 定位到上一次的完整分析，而不是从零开始？
- 我们说不确定的事情，是真的"不知道"，还是"知道但我们不想写"？后者更危险 — 遗漏已知风险不是诚实，是埋雷。
- 元学习捕获的东西，下次做 8D 时会用上吗？如果会，怎么用 — 是改流程、改模板、还是改培训？

## 常见跳步模式和阻止策略

**跳步 1：为结案而结案**
表现：D6 数据还没出就结案，把"没再投诉"当成"措施有效"。
阻止策略：D6 量产验证数据必须附件提交，无数据不审批。确有业务理由需提前关的，必须在不确定性声明里显式标注。

**跳步 2：团队自评全是 Excellent**
表现：所有步骤打最高分，没有一条改进建议。
阻止策略：至少标注 1-2 个"下次可以做得更好"的步骤，即使打 Excellent 也要写具体证据。

**跳步 3：没有 Truth Chain**
表现：报告各部分都有内容，但没有因果链串联，读者需要自己拼图。
阻止策略：强制输出单行 Truth Chain（现象->根因->流出点->措施->验证->经验），每段一句。某段填不出来就说明对应 D 步骤没做扎实。

**跳步 4：不确定性不声明**
表现：报告中不出现"不确定""可能""待验证"的表述，假装一切已完全解决。
阻止策略：模板强制"不确定性声明"栏，留空或写"无"视为不完整。至少列出验证局限性（样本量、周期、边界覆盖度）。

## 收敛过早的警告信号

- 结案审批在 D6 量产数据出来之前就完成了。
- 团队自评全 Excellent，没有一条改进建议。
- Truth Chain 缺某一段（最常见是缺 Escape Point 那段或经验那段）。
- 不确定性声明为空白，或写"无"。
- 元学习栏写"本次 8D 做得很好，没有需要改进的流程"。
- 管理层签核只有签名没有意见，或者签核人不具备技术判断能力。
- 报告归档路径不明确 — 问"报告在哪"需要超过 2 分钟才能找到。

## Excellent 级别标杆

- Truth Chain 完整编码，从现象到经验一句贯通，每段可追溯至报告对应章节。
- 团队自评有 Insufficient/Basic/Excellent 的分化，Insufficient 项有明确的补强计划和负责人。
- 不确定性声明诚实地列出了验证局限性（样本量、边界覆盖不足、长期数据缺失），并给出了每个不确定项的管理策略（定期复检 / 设计余量 / 持续监控）。
- 元学习至少捕获了一个 8D 流程改进点（如"D4 应该先列候选原因再逐一排除，而不是直接跳到最可能的原因"），并有人跟踪落地。
- 管理层签核附具体意见，不只是"同意结案"。
- 完整报告与附件按编号体系归档，从搜索到检索用时不超过 2 分钟。
- 复检计划已定义：3-6 个月后复查关键指标，复查结论记录在册。
