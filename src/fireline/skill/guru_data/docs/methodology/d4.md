# D4 — 根因分析

## 行业最佳实践

D4 是 8D 的心脏。前三个 D 是为了把问题描述清楚、把团队拉齐、把出血止住——所有这些铺垫都是为了 D4 能精准地找到根因。ASQ 和 Ford 8D 手册对 D4 的要求是双线并行：找到问题为什么发生（Occurrence Root Cause），找到问题为什么流出了（Escape Point Root Cause）。两条线缺一不可——只找发生原因不找流出原因，下次换个方式同样的问题还会漏出去。

### 双根因体系：TRC + MRC

行业标准要求区分 Technical Root Cause（技术根因）和 Managerial Root Cause（管理根因）：

- **TRC**：物理/技术层面的直接原因。为什么这个东西坏了？追溯到物理机制、设计假设或控制回路层面的谬误。
- **MRC（四要素）**：Management System（管理体系-有没有标准？）、Business Processes（业务流程-标准有没有被执行？）、Personnel（人员-执行的人有没有能力？）、Organization（组织-有没有激励或约束机制？）。

技术根因不解释为什么会发生——解释的是"物理世界哪里出错了"。管理根因解释的是"为什么人没能阻止这个物理错误发生或流出"。大多数 8D 只找到 TRC 就停了，这是最常见的深度不够。

### 双 5-Why 链

对发生原因拉一条 5-Why 链，对流出原因单独拉另一条 5-Why 链。两条链的起点不同、路径不同、终点不同。**禁止停在"操作员失误""供应商问题""来料不良""培训不足""设备坏了"这五类假根因上**——这些都是表象或责任归属，不是可行动的系统性问题。

真正到位的 5-Why，最后一层的答案应该是：某个具体的控制点（设计评审/过程检验/设备点检/变更审批/技能认证）没有起作用，并且能说出为什么没起作用。

### 核心分析工具链

**Is/Is-Not 缩小搜索范围**：D2 定义的边界在 D4 变成搜索约束。Is-Not 可以直接排除搜索方向——"这个批次之后没有不良品"意味着要在这个批次的时间点附近找变化。

**差异与变化分析（Difference & Change Analysis）**：对比好品和坏品之间的差异点，追溯这些差异是什么变化引入的、什么时候引入的、谁引入的。变化不一定是原因，但变化点是搜索方向。

**主动验证（Turn Root Cause On/Off）**：这是区分"可能原因"和"确认原因"的黄金标准。如果你认为根因是某个参数漂移，把参数调回去看看问题是不是消失了；再把参数漂移回异常值看看问题是不是复现了。能开关的根因才叫确认根因。

**鱼骨图 6M + 系统交互**：6M（人/机/料/法/环/测）是候选原因的搜索框架，但行业教训是：单一原因很少是真相。大多数质量问题是多个原因的系统交互——某个控制点弱化了但没完全失效，直到另一个条件变化使弱点变成失效。鱼骨图上要找的不只是单点原因，还有原因之间的交互关系。

**反事实测试（Counterfactual Testing）**：对每个候选根因问"如果这个原因不存在，问题还会发生吗？"如果会，这个原因至少不是唯一原因，或者不是主要原因。反事实测试用来排除假根因非常有效。

### 第一性原理分解

D4 的追问需要穿透四层：

1. **物理机制层**：失效的实际物理/化学过程是什么？焊点开裂是疲劳还是过载？疲劳的应力源是什么？应力是怎么传导到焊点上的？
2. **设计假设层**：设计时对这个物理机制做了什么假设？假设是否成立？假设在什么条件下被突破了？
3. **控制回路层**：设计假设被突破时，应该有哪个控制点检测到？那个控制点为什么没有检测到？是控制点不存在、控制点失效、还是控制点被绕过了？
4. **组织层**：为什么控制回路没有被维护好？是资源问题、优先级问题、知识问题、还是激励问题？

## 探索方向矩阵

D4 是所有 D 中探索方向最多的。至少需要覆盖 9 个方向：

| 方向 | 示例问题 |
|------|---------|
| 物理失效机制 | 失效在本体上是什么物理/化学过程？是逐步退化还是突然发生？失效界面在哪里？ |
| 差异分析 | 好品和坏品的可测量差异是什么？尺寸/电性/材料/工艺参数——哪一项有统计显著差异？ |
| 变化点追溯 | 什么时候变坏的？时间点上有什么可控事件（换批/换人/修机/改参数/换供应商）与之重合？ |
| 发生原因 5-Why | 从物理机制开始向下追：为什么这个机制被触发了 → 为什么触发条件存在 → 为什么没人阻止触发条件 → ... |
| 流出原因 5-Why | 这个问题为什么没有被检测到？最早应该拦住它的控制点在哪里？那个控制点为什么没拦住？控制点失效的原因是什么？ |
| 设计假设检查 | 设计规格是否覆盖了实际使用条件？设计裕度是否足够？失效是否在设计 FMEA 中已被预见到？ |
| 6M 系统交互 | 不是一个原因，而是哪两个或三个因素的组合？人+法？机+料？环+测？组合条件是必要条件还是充分条件？ |
| 主动验证 | 能把怀疑的根因打开和关掉吗？怎么做物理实验？怎么做数据分析？不做实验能不能做历史数据回溯验证？ |
| 反事实测试 | 如果这个根因不存在，问题会不会仍然发生？如果会，还有哪个原因没找到？ |
| 管理根因四要素 | 管理体系有标准吗？流程执行了标准吗？人员有能力执行吗？组织有正确的激励吗？四个要素逐个检查。 |
| 历史同类问题 | 以前有过类似问题吗？当时的纠正措施为什么没防住这次的问题？是措施失效了还是措施被撤销了？ |
| FMEA 反查 | 这个失效模式在 PFMEA/DFMEA 里有没有？RPN 是多少？当时的预防/探测措施实际起作用了吗？ |

## 常见跳步模式和阻止策略

**跳步模式 1：5-Why 停在第二层。** "为什么短路？因为来料有毛刺。为什么来料有毛刺？供应商模具老化。结论：要求供应商修模。"识别方式：5-Why 链小于 3 层，或最后一层指向外部（供应商/客户/天气）。阻止方式："供应商模具为什么老化了？老化了为什么没检测到？检测了为什么放行了？我们需要追到我们自己的控制点失守。"

**跳步模式 2：只有 TRC 没有 MRC。** 找到了焊点开裂的技术原因就认为做完了。识别方式：树图上完全没有人/流程/组织/管理的内容。阻止方式："技术原因说明了为什么会坏，但没有说明为什么这个人/这个流程/这个系统允许它坏掉并流出。下一次类似的失效模式一样会流出。"

**跳步模式 3：找到了发生原因但没找到流出原因。** 只做了一条 5-Why。识别方式：Escape Point 栏空白或写"已找到发生原因"。阻止方式："找到为什么发生很好。现在换一个方向：从检测端往回看——这个问题最早在哪个工序就该被拦下来？为什么没拦住？"

**跳步模式 4：原因没有做主动验证。** 找了一堆可能原因，选了最合理的一个当结论。识别方式：问"你怎么排除其他候选原因的"时回答"经验判断"。阻止方式："我们需要把怀疑的根因打开和关掉，看问题是不是随之出现/消失。没有验证的根因只是有根据的猜测。"

**跳步模式 5：用措施反推原因。** "根因是培训不足，所以加强培训。"识别方式：根因描述和措施描述形成了循环论证。阻止方式："培训不足是措施缺口，不是根本原因。为什么培训不足？是内容不对、是没人教、是教了没考核、还是考核了没关联绩效？"

## 收敛过早的警告信号

- 5-Why 链少于 3 层就停了。
- Escape Point 没有找到，或找到了但没做独立的 5-Why。
- 候选原因少于 3 个——只考虑了一个方向就下结论。
- 没有做主动验证（物理实验或历史数据回溯）。
- MRC 四要素只填了"流程"或"培训"，其他空白。
- 候选根因没有用概率表追踪——只有一个候选根因被认真考虑，其他被忽略。正确做法是维护贝叶斯概率表，每轮证据更新所有假设的概率（见 `docs/patterns/bayesian-hypothesis.md`）。

## 贝叶斯假设追踪

在 D4 S1 后初始化候选根因概率表。每轮 S2/S3 后更新概率。决策规则：

- 最高概率 < 40% → 继续发散，可能遗漏了重要方向
- 最高概率 40-70% → 深度收敛，追物理机制
- 最高概率 > 70% 但反事实未过 → 集中反事实挑战
- 最高概率 > 80% 且反事实通过 → 可进 S5

详细规则见 `docs/patterns/bayesian-hypothesis.md`。
- 根因结论里有"操作员失误""来料不良""设备故障"等禁止停留的答案。
- 用户表现出"差不多就是它了"的态度但没有验证数据支持。
- 控制点判断缺失——说不出来到底是哪个控制点失守、为什么失守。

## Excellent 级别标杆

- 双 5-Why 链完整：发生原因链至少 5 层，流出原因链至少 3 层，两条链都追到了具体控制点失守。
- 根因经过了主动验证：有实验数据或历史回溯数据，能证明根因开则问题现、根因关则问题消。
- 至少 3 个候选原因被系统排除（不是被忽略），每个排除有依据。
- TRC 追到了物理机制层，不是只说"焊接不良"而是说"IMC 层厚度 4.2μm 超出规格上限 3.0μm，导致焊点脆化"。
- MRC 四要素齐全：管理体系、业务流程、人员、组织四个维度都有分析。
- 差异分析 + 变化点分析有时序依据（变更记录/参数趋势图/批间对比数据），不是纯访谈。
- 设计假设检查已完成：确认设计规格和实际使用条件之间的差距，FMEA 已反查。
- 反事实测试已执行：对确认的根因做了"去掉后问题会不会仍然发生"的逻辑检验。
- 探索方向覆盖 9 个以上，各方向有记录（即使排除方向也有排除理由）。
- 整个 D4 的结论可以不经翻译直接输入 D5 的纠正措施设计。
