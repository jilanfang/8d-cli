# D6 — 验证

## 行业最佳实践

D6 验证与 D5 验证的关键区别：D5 是实验室/试产条件下的功能确认（"这个措施在受控环境里能不能起作用"），D6 是量产条件下的统计证明（"这个措施在真实生产环境中持续起不起作用"）。大量 8D 在此翻车 — D5 过了就宣布胜利，D6 草草收场。

行业标准要求四项强制证据：

1. **统计能力证明**：关键质量特性（CTQ）的过程能力指数 Cpk 大于等于 1.33。仅凭"最近没再收到投诉"不是验证。必须有数据：缺陷率趋势、SPC 控制图趋势、Cpk 前后对比。如果样本量不足以计算 Cpk，至少要有假设检验（如二项检验）证明改善前后差异统计显著。
2. **Red Rabbit 测试**（也称为"故意放蛇"测试）：人为制造已知缺陷品投入产线，验证检测系统能否捕获。这是验证 Escape Point 措施是否有效的唯一可靠方法。如果 Red Rabbit 没被拦住，说明流出控制点仍然失守。
3. **边界条件验证**：不仅要验证正常工况，还要验证边界 — 换型首件、设备重启后第一件、极限公差组合、新人操作、夜班/交接班等。大多数措施在正常条件下没问题，一到边界就失效。
4. **ICA 退出确认**：D3 的临时围堵措施只能在 PCA 确认有效后才能撤。撤出要有证据 — 通常要求 PCA 在量产条件下至少运行一段连续时间（如 30 天或 3 个批次）且无异常，才能逐步撤 ICA。不允许 PCA 和 ICA 同日切换。

## 探索方向矩阵

|探索方向|示例问题|
|------|------|
|统计方法选择|数据是连续型还是离散型？该用 Cpk 还是 Ppk？样本量不够怎么办 — 能不能用假设检验替代？谁来判定"统计显著"？|
|Red Rabbit 设计|故意缺陷品的规格怎么定 — 刚好在边界上还是明显超标？放进去多少件、覆盖哪些班次/机台？如果 Red Rabbit 被拦住了但真品没拦住怎么办？|
|边界条件覆盖|验证计划覆盖了哪些边界条件？换型、换班、换人、换料批次、设备维护后、长假后复产 — 漏了几个？|
|验证周期与样本|需要多少个批次/多少天才能有信心说"措施有效"？这个数字怎么来的 — 是拍脑袋还是基于缺陷率和置信度算出来的？|
|ICA 退出策略|ICA 什么时候可以撤？是一下全撤还是逐步撤？撤的顺序是什么？撤的过程中用什么监控 — 万一 PCA 失效怎么回滚？|
|验证失败的预案|如果 D6 数据不通过怎么办？回 D5 重选措施还是回 D4 重新找根因？谁来判断、触发条件是什么？|
|客户侧验证|客户要不要参与验证？客户端的验证条件（装机环境、使用场景）和厂内验证条件有什么不同？|

## 第一性原理锚点

D6 的本质问题是：**我们有没有在真实生产噪声中，用数据而非信念证明因果链已经断了？**

追问到这个层才算到位：

- D5 验证是在降噪环境里做的，D6 验证要把产线的所有噪声加回来 — 班次差异、人员差异、来料波动、设备漂移、环境变化。措施在这些噪声下还能稳得住吗？
- 缺陷率归零不等于措施有效。小概率事件在短期内不出现是正常的。你怎么区分"措施有效所以没再发生"和"运气好所以还没遇到"？
- ICA 撤了以后，如果 PCA 顶不住，需要多长时间才能发现？发现之前会漏出去多少不良品？

## 常见跳步模式和阻止策略

**跳步 1：用 D5 结果替代 D6 验证**
表现：试产 50 件没问题，直接宣布措施有效，量产不再跟踪。
阻止策略：D6 验证方案必须在 D5 阶段就写好，明确量产条件下的验证周期、样本量和判定标准。试产通过只能算 D5 close，不能算 D6 close。

**跳步 2：没有统计证据**
表现：验证结论写"验证通过""客户没再投诉""产线反馈没问题了"。
阻止策略：要求验证结论至少包含一种统计证据 — SPC 图前后对比、缺陷率前后对比加假设检验、Cpk 计算。如果没有数据能算，说明抽样方案没设计好。

**跳步 3：ICA 与 PCA 同日切换或 ICA 提前撤出**
表现：PCA 一上线就把 ICA 撤了，或者 PCA 还没验证完 ICA 就停了。
阻止策略：ICA 撤出必须有书面批准，附 PCA 稳定运行的证据（如连续 N 批次无异常数据）。ICA 撤出后要有回溯监控期。

**跳步 4：只验证正常条件，跳过边界**
表现：验证只在白班、老员工、常规批次上做。
阻止策略：验证计划强制覆盖至少 3 个边界条件：换型首件、新人操作、极端公差组合。如果这 3 个条件没有覆盖，验证不完整。

## 收敛过早的警告信号

- 验证结论全是文字描述，零统计数据。
- 没有 Red Rabbit 测试记录或等价检测系统验证。
- 验证样本量小于 30 且没有统计理由。
- ICA 撤出时间和 PCA 上线时间是同一天，没有重叠期。
- 边界条件一个都没测，验证范围只覆盖"正常生产"。
- 验证失败的预案缺失：没写"如果不过怎么办"。
- 客户侧没有参与验证，但问题是在客户那里发现的。

## Excellent 级别标杆

- Cpk 大于等于 1.33（或等价统计证据），附前后对比图表和原始数据。
- Red Rabbit 测试 100% 被检测系统捕获，测试设计文档记录了缺陷品规格、投放方式、覆盖范围。
- 边界条件验证覆盖至少 5 类：换型、换班/换人、设备重启、极限公差、来料批次切换。
- ICA 按阶段撤出，PCA-ICA 重叠时间明确，回滚方案已定义。
- 验证数据包含足够长的运行时间（如 30 天或 3 个量产批次，取较长者），能区分"运气"和"效果"。
- 客户侧验证结果与厂内结果一致，客户书面确认。
