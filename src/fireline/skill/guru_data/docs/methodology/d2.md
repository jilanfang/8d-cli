# D2 — 问题定义

## 行业最佳实践

ASQ 和 Ford 8D 手册将 D2 定位为"把客户投诉转化为可调查的工程问题"。D2 是整个 8D 的底座——问题定义偏一度，D4 根因分析偏十度。行业里有一句话：一个定义清楚的问题，已经解决了一半。

D2 的核心工具链：

**Is/Is-Not 矩阵**是最强的问题边界工具。用五个维度（What/Where/When/Who/HowMany）把"问题是什么"和"问题不是什么"同时写清楚。Is-Not 的价值常常大于 Is——知道哪些批次没受影响、哪个工序不会复现、哪个客户没有投诉，比只知道"出事了"更能锁定范围。

**5W2H + SMART 问题陈述**：What（什么失效）/ Who（谁发现的）/ Where（在哪个工序/位置）/ When（什么时候）/ Why（为什么是问题——影响量化）+ How（怎么发现的）/ How Many（多少数量）。最终产出必须满足 SMART：Specific, Measurable, Actionable, Relevant, Time-bound。不允许"最近质量不太稳定"这种不可调查的描述。

**Gemba（现场现物）**：问题定义不能只在会议室完成。必须有人去实际地点看实际产品。没有 Gemba 的 D2 只是"听说的 D2"。

**CATWOE 利益相关方分析**：Customers（受害者是谁）/ Actors（谁在执行相关操作）/ Transformation（输入到输出发生了什么变化）/ Worldview（各方怎么看待这个问题）/ Owner（谁能改变这个系统）/ Environment（外部约束是什么）。CATWOE 确保问题定义不只是工程技术视角，还有组织和人的视角。

**事实/推断/假设三级分离**：这是第一性原理思维的基础。事实是可直接观察或测量的（"5V 引脚在负载下测得 0V"）。推断是基于事实的逻辑延伸（"因为测到 0V 所以可能是短路"）。假设是尚未验证的猜想（"可能是操作员没按 SOP"）。D2 要求把这三层分开写，不允许把推断当事实用。

## 探索方向矩阵

| 方向 | 示例问题 |
|------|---------|
| 客户视角还原 | 客户实际看到/测量到什么？复现条件是什么？影响产线还是终端用户？ |
| 内部映射 | 这个失效对应内部哪个工序/测试项/控制点？内部的不良代码是什么？ |
| Is/Is-Not 边界 | 哪些产品/批次/产线/客户/时间段受影响，哪些没受影响？ |
| 时间线重建 | 什么时候开始出现？是突然恶化还是逐渐漂移？与任何变更事件（人/机/料/法）时间重合吗？ |
| 影响量化 | 不良数量、不良率、客户停线时间、产生的费用、潜在召回风险。 |
| Gemba 验证 | 有没有人去现场看了实际产品？在会议室描述的和在现场看到的是一致的吗？ |
| 利益相关方 | 谁受影响、谁在操作、谁在决策、谁在旁观？各方对问题的描述一致吗？ |

## 第一性原理锚点

D2 的追问应该停在这一层：**"这个问题在物理世界里的精确边界在哪里——不是'感觉哪些有问题'，而是'确认哪些有、确认哪些没有、确认边界外最近的一个 OK 样本在哪里'。"**

追问路径：

- 对每一个 Is 陈述，追问对应的 Is-Not："这个时间段有问题——相邻时间段确认没问题吗？这个批次有问题——相邻批次确认没问题吗？这个客户投诉了——同批其他客户确认没投诉吗？"
- 对每一条"事实"，追问证据来源："这个事实是谁看到的？在哪里看到的？什么时候看到的？有没有记录？如果没有记录，为什么认为是事实？"
- 把 SMART 检查套到问题陈述上：Specific（模糊词句都消除了吗）/ Measurable（有数值吗）/ Actionable（调查团队能据此行动吗）/ Relevant（和客户投诉直接相关吗）/ Time-bound（有人时间边界吗）。

如果问题陈述里还有"可能是""感觉是""据说"，D2 还没做完。

## 常见跳步模式和阻止策略

**跳步模式 1：问题描述里混入根因。** "USB 无输出是因为焊接不良"——后半句是根因假设，不是问题定义。识别方式：描述里出现了"因为""所以""导致"。阻止方式："我们先只描述眼睛看到的东西。为什么会这样，到 D4 再追究。"

**跳步模式 2：跳过了 Is-Not。** 用户只说了什么有问题，没说什么是好的。识别方式：Is-Not 栏为空或只有一条模糊信息。阻止方式："知道什么没问题，常常比知道什么有问题更能定位范围。D3 围堵需要这个边界——不知道边界就不知道该围堵多少。"

**跳步模式 3：没有去现场。** 所有信息来自邮件或微信截图。识别方式：问"你亲眼看过失效件吗"时回答"没有，但生产部说……"。阻止方式："转述的信息至少经过一层衰减。D2 必须有至少一个人去现场看了实际产品。我们可以先暂停，你去趟车间再回来填。"

**跳步模式 4：影响量化用形容词。** "影响比较大""客户很生气"。识别方式：没有数字。阻止方式："不良率是多少？停线了多久？涉及多少费用？我们需要数字，不是为了写报告，而是为了判断 D3 围堵的紧迫度和 D5 永久措施的资源投入。"

## 收敛过早的警告信号

- Is-Not 矩阵少于 3 个维度有有效信息。
- 事实清单里包含"可能是""估计""大概"开头的信息——这些是推断或假设，不是事实。
- 没有去过现场（Gemba 未执行），所有信息来自二手转述。
- 问题陈述不能通过 SMART 检查——缺数字、缺时间、缺具体对象。
- 利益相关方只有客户一方视角，没有操作员/检验员/供应商的描述。
- CATWOE 分析缺失——只看了技术维度，没看人和组织维度。

## Excellent 级别标杆

- Is/Is-Not 矩阵覆盖全部五个维度（What/Where/When/Who/HowMany），每个维度至少有一条 Is 和一条 Is-Not。
- 问题陈述通过 SMART 检查，可以不经翻译直接作为 D4 根因分析的输入。
- 事实/推断/假设三级明确分列，事实有证据编号或来源标注。
- Gemba 已执行，有现场照片/测量记录/实物观察笔记，不只有二手信息。
- CATWOE 六个要素都有人对应，不同利益相关方的描述差异已记录（即使暂时解释不了）。
- 影响量化具体到数字：不良件数、不良率、涉及金额或客户停线时间。
- 时间模式已识别并用于指导 D4 根因搜索方向（见 `docs/patterns/temporal-patterns.md`）。
