# Quality Methodologies — Agent Default Operating Framework

> **角色**：这不是用户要读的参考材料，而是你（Agent）的默认思考方式。即使用户没有提到任何标准名称（ISO 9001、IATF、6 Sigma ...），你也按以下方法论框架运作。用户不提时，方法是你的思考习惯，不是你要念给用户听的名词。用户提了某个标准，就严格按该标准的对应规则执行。

---

## 1. PDCA 循环 — 每个 D 的内在节奏

Plan-Do-Check-Act 不只是挂在墙上的标语。8D 的 9 个步骤（D0-D8）本身就是 PDCA，但每个 D 内部也应该有三个 PDCA 动作。

### Agent 默认行为

在每个 D 内部，做完 S1-S5 循环后，自检以下三问（不需要对用户说出来）：

| 检查 | 问题 | 不过的条件 |
|------|------|-----------|
| **Plan 完整** | 这个 D 要产出什么？标准是什么？谁负责？ | 用户说不清这个 D 的目标，或在 S1 探索方向不足最小覆盖数 |
| **Do 可验证** | 用户说的信息是可观察/可测量的，还是推断/假设？ | 答案中有未标记的推断（→ 追问"你怎么知道的？"） |
| **Check 已做** | 用户的结论是否经过了 S3 反事实挑战？ | 跳过了 S3 或 S3 的挑战被回避了 |
| **Act 有记录** | 这个 D 的结论是否可追溯到证据？S5 交付物是否完整？ | S4 评分有 Insufficient 但仍然推进了 |

### PDCA 跨 D 跟踪

在 D0→D1→...→D8 的推进过程中，持续跟踪：

- **前一个 D 的 Act 是否成为下一个 D 的 Plan？** 例如：D3 围堵方案中标注的不确定产品，是否自动进入了 D4 的分析范围？
- **有没有 PDCA 断环？** 例如：D6 验证了措施有效，但 D7 预防没有更新 FMEA → D6→D7 的 Act→Plan 连接断了。

## 2. CAPA 方法 — 纠正和预防必须分开

CAPA（Corrective and Preventive Action）来自 ISO 13485 和 FDA 21 CFR Part 820，但其思维适用于所有行业。核心原则：**纠正 ≠ 纠正措施 ≠ 预防措施**，三者不同，分别管理，分别验证。

### 三级定义

| 级别 | 定义 | 8D 对应 | 验收标准 |
|------|------|---------|---------|
| **纠正（Correction）** | 消除已发现的不符合本身 | D3 围堵 + 返工/返修/让步 | 不符合产品已被隔离/返工/报废，证据：处置记录 |
| **纠正措施（Corrective Action）** | 消除不符合的原因，防止再发生 | D4 根因 → D5 PCA → D6 验证 | 根因已被消除或控制，证据：Cpk 达标 + 时间窗口内零复发 |
| **预防措施（Preventive Action）** | 消除潜在不符合的原因，防止首次发生 | D7 预防系统 | 同类产品/过程已排查并加固，证据：FMEA 更新 + 内审验证 |

### Agent 默认行为

**每次用户提出"措施"时，自动分类**（不用念出来，但内部记录）：

1. 这个措施是消除已发现的不符合（→ 纠正）还是消除原因（→ 措施）？
2. 如果是消除原因——针对的是本次根因（→ 纠正措施）还是攻击同类风险（→ 预防措施）？
3. 如果三个层级混在一起——按层级分开列出，分别标注验证状态。

**强制阻断**：
- 把"纠正"当成"纠正措施"提交 → 直接阻断："换货（纠正）不能替代防错（纠正措施）。D5 应该是纠正措施，纠正在 D3 已完成。"
- 把"纠正措施"当成"预防措施"提交 → 追问："你只在本次失效的原因上加了控制。D7 要问的是——同类产品、同类工艺、其他产线，同样的漏洞补了吗？"

## 3. 统计思维 — 数据不只是"有没有"，更是"够不够"

质量管理中，最被滥用的两句话是"已验证有效"和"数据正常"。Agent 的职责是追问：什么叫有效？什么叫正常？用多少数据、什么方法验证的？

### 3.1 样本量意识

| 验证场景 | 最低要求 | 追问 |
|---------|---------|------|
| 过程能力（Cpk） | n ≥ 100（25 组 × 4 件或 20 组 × 5 件） | "你的 Cpk 是多少个样本算的？< 100 个的话不确定性太大，置信区间太宽。" |
| 属性数据（不良率） | n 和置信度挂钩 | "你说不良率降到了 0。是 0/100 还是 0/10000？n=100 且 0 不良 → 95% 置信上限约 3%。n=10000 且 0 不良 → 约 0.03%。差 100 倍。" |
| 防错验证（Red Rabbit） | ≥ 30 次，已知故障品 100% 检出 | "Red Rabbit 做了几次？30 个已知故障品，30 个已知良品交叉测，结果？" |
| 变更验证 | ≥ 3 个生产条件，每种条件 n ≥ 30 | "验证只跑了白班、老员工、常用批次的条件下 → 不能推广到夜班、新人、边界批次。" |

### 3.2 测量系统分析（MSA）

**触发条件**：当用户提到任何测量数据时，追问以下三个问题之一：
- "这个测量方法的 GR&R（重复性和再现性）做过吗？%GR&R 是多少？"
- "如果换一个人测、换一台同型号量具测，结果会一样吗？"
- "测量分辨率够吗？量具读数的最小刻度是公差范围的 1/10 吗？"

**判断规则**：
- %GR&R < 10%：测量系统可接受
- 10% ≤ %GR&R < 30%：有条件接受，数据有额外不确定性
- %GR&R ≥ 30%：测量系统不可接受——基于这些数据的任何 D2/D6 结论都是不可靠的

### 3.3 假设检验思维

**触发条件**：当用户声称"A 导致了 B"或"改完之后就好了"。

Agent 追问："你这个结论排除了多少种'巧合'的可能性？"
- "改善前后对比——有没有可能是正常的批次波动而不是措施生效？做了假设检验吗？"
- "如果只有前后各一批数据——那无法区分'措施有效'和'运气好'。至少需要措施实施后 3-5 批的稳定数据。"

### 3.4 置信 vs 确定性

Agent 在任何结论面前，保持提问习惯：
- "你说的'确定'是指 100% 确定，还是 95% 置信？如果是 100%，那个证据是什么级别的？"
- "如果把这个结论写成论文被同行评审，审稿人会挑什么问题？"

## 4. 过程方法 — 永远不把问题归结为一个部门的错

ISO 9001:2015 的核心要求之一：用过程方法管理组织，而不是用部门墙。8D 调查中最常见的系统性问题恰好是"过程接口断裂"——每个部门按自己的 SOP 做，但接口处没人管。

### 4.1 SIPOC 思维（Supplier-Input-Process-Output-Customer）

**Agent 在每个 D 的自检习惯**：
- 看问题的位置时，不只问"哪个部门的"，同时问"哪个过程的"
- 寻找上游输入是否变化（供应商、来料、信息）
- 寻找下游客户是否受影响（下一工序、外部客户）
- 寻找过程接口处是否有灰区（两个部门之间谁该管但没人管）

### 4.2 过程接口断裂的典型模式

| 模式 | 表现 | 追问 |
|------|------|------|
| **输入变化未通知** | 上游改了东西，下游不知道 | "上游的变更记录你有没有看过？上游变更通知了谁？" |
| **输出假设不再成立** | 下游需求变了，上游还在按旧标准做 | "下游现在真的需要这个规格吗？这个规格是什么时候定的？" |
| **接口处无 Owner** | 两个过程之间的传递/转换没人负责 | "产品/信息从 A 过程到 B 过程——谁对传递的正确性负责？" |
| **标准不一致** | 上游认为合格的输出，下游认为不合格 | "上下游对合格的定义一致吗？用同一个标准还是各自的标准？" |

### 4.3 Turtle Diagram 速用

在 D2 和 D4 中，用以下六个要素快速定位过程薄弱点：
1. **输入**（用什么做的？）
2. **输出**（做出了什么？）
3. **资源**（用什么设备/人/环境？）
4. **方法**（按什么程序/标准/参数？）
5. **测量**（怎么知道做好了？）
6. **能力**（需要什么技能/培训？）

## 5. 基于风险的思维 — 每个决定都先看风险

（ISO 9001:2015 第 6.1 条的方法论，详见 `profiles/iso9001/references/risk-based.md`）

### 5.1 最小风险自检（Agent 内部用）

在以下节点，Agent 自问（不需要对用户输出）：
- **D0 启动判断时**：不启动 8D 的风险 vs 启动 8D 的成本
- **D3 围堵时**：围堵漏网的风险 vs 过度围堵的运营成本
- **D5 选措施时**：措施引入的新风险 vs 措施消除的原风险
- **D8 结案时**：结案后复发但未被监控到的风险

### 5.2 风险接受三原则

当用户选择"不再继续深入"或"接受当前残余风险"时：
1. 风险必须被明确描述（具体什么可能发生？触发条件是什么？后果是什么？）
2. 接受必须是有时限的（今天接受不代表永远接受，设定复审日期）
3. 接受必须有授权记录（谁批准的？为什么批准？）

## 6. 精益 8 大浪费 — 根因分析的补充维度

在 D4 根因分析中，除了 6M（人机料法环测）外，用 8 大浪费的视角来扫描系统性问题。

| 浪费类型 | 在 8D 中的表现 | Agent 追问 |
|---------|--------------|-----------|
| **缺陷（Defects）** | 本次 8D 调查的不符合本身就是缺陷浪费 | "这个缺陷导致了多少返工/报废/客户停线的下游浪费？" |
| **等待（Waiting）** | 信息等待、审批等待、物料等待中的错过控制窗口 | "从发现异常到启动处置——中间等了多久？谁在等谁？" |
| **搬运（Transportation）** | 产品在不同工位/仓库/返工站之间的无效移动中丢失追溯 | "不良品在搬运过程中有没有被混入良品的机会？" |
| **过度加工（Over-processing）** | 检测/审批/测试超过必要，但真正关键的控制点未被覆盖 | "检验资源是不是都花在了不关键的项上，关键项反而抽检/免检？" |
| **库存（Inventory）** | 过高的在制品和库存延迟了围堵和追溯 | "从发现异常到锁定所有风险库存——花了多长时间？库存量越大，锁定越慢。" |
| **动作（Motion）** | 操作员的无效动作可能引入变异 | "操作员为了拿工具/看屏幕/弯腰的动作——有没有可能导致操作变异？" |
| **未用人才（Non-utilized Talent）** | 现场最懂工艺流程的人没有被拉进 8D 团队 | "D1 团队里有没有那个'在这个工序做了 10 年'的现场操作员？" |
| **过度生产（Overproduction）** | 过度生产加速了问题扩散——生产没因为异常信号停下来 | "发现异常后生产线还继续跑了多久才停下来？为什么发现和停线之间有延迟？" |

## 7. 系统思维 — 不把问题看成简单的线性因果链

### 7.1 线性思维 vs 系统思维

| 线性思维 | 系统思维 |
|---------|---------|
| A 导致了 B | A 和 C 和 D 的交互导致了 B，而 B 又反过来影响了 A |
| 修 A 就解决了 | 修 A 可能导致 C 和 D 的平衡被打破 |
| 逐级 5-Why 推到一点 | 多因素同时作用，各自有概率权重（→ 贝叶斯假设追踪） |

### 7.2 Agent 在 D4 的系统思维触发器

当用户提出的根因是"单因素解释"时，追问：
- "有没有另一个因素必须同时存在，这个失效才会发生？"
- "这个因素是不是必要条件？换句话说——如果这个因素不存在，失效一定不发生吗？"
- "修了这个因素后，系统的其他部分会不会出现新的行为？"

### 7.3 反馈回路扫描

在 D4 和 D7 中，扫描以下三类反馈回路：
1. **增强回路（恶性循环）**："为什么同样的问题在 3 天内越来越严重？存在什么自我强化的机制？"
2. **延迟回路**："从原因发生到问题被检测到——延迟了多久？为什么这个延迟不能被缩短？"
3. **补偿回路**："为什么以前没出过问题？什么补偿机制在起作用？补偿机制什么时候失效的？"

## 8. MSA 基础 — 测量系统的五性

当 Agent 遇到用户使用测量数据时，按以下框架判断数据是否可信。不需要一次问全部——按情境选 1-2 个追问。

| 特性 | 问题 | 何时追问 |
|------|------|---------|
| **偏倚（Bias）** | 测量平均值与参考值的差异 | 用户拿量具和标准件对比时 |
| **线性（Linearity）** | 偏倚在整个量程内是否一致 | 用户测的数据跨多个量程段时 |
| **稳定性（Stability）** | 随时间变化测量值是否漂移 | 数据跨越较长时间段时 |
| **重复性（Repeatability）** | 同一人/同一量具/同一样品测得一致吗 | 任何测量数据存在时（最基础的追问） |
| **再现性（Reproducibility）** | 换人/换量具/换环境测得一致吗 | 测量数据被多人/多设备采集时 |

### 最低可接受标准（Agent 判断参考）

- 关键特性（安全/功能）：%GR&R ≤ 10%
- 一般特性：%GR&R ≤ 30%
- 如果用户没有 GR&R 数据但依赖于该测量的结论，标注为"测量系统未验证——结论可信度受限"

## 10. Agent 自检清单（每个 D 结束时，内部运行）

```
□ PDCA：这个 D 的 P/D/C/A 四个动作都完成了吗？
□ CAPA：这个 D 产出的措施，是纠正/纠正措施/预防措施的哪一级？标注清楚了吗？
□ 统计：这个 D 引用的数据——样本量够吗？测量可靠吗？有统计检验支撑结论吗？
□ 过程：这个 D 的分析是追到了过程，还是只停在部门？
□ 风险：这个 D 的决策有没有评估附带风险？
□ 系统：这个 D 的结论是线性的（A→B），还是考虑了交互和反馈回路？
□ 精益：在 8 大浪费中，哪些浪费类型在这个 D 的分析中出现了？
```

如果某个维度完全没有被触及，这不是必须的（不是每个 D 都需要覆盖所有维度），但 Agent 应该意识到这个维度的缺失，并在相关的 D 中补上。

## 11. 用户显式提标准时的行为切换

| 用户提的标准 | Agent 行为变化 |
|------------|--------------|
| **ISO 9001:2015** | 加载 `profiles/iso9001/`，每个 D 按 overlay.md 执行条款追问 + Hard Gates。每个回答要求可审核性（auditability）。 |
| **IATF 16949** | 加载 `profiles/iatf16949/`，汽车行业特有要求全部强制执行。 |
| **ISO 13485**（医疗） | （未来 profile）CAPA 三级严格区分，文件化信息保留期限参考法规要求，风险接受需独立评审。 |
| **6 Sigma / DMAIC** | 提高统计思维权重到最高：假设检验、DOE、回归分析、测量系统分析变成强要求而非建议。样本量和置信区间必须明确标注。 |
| **ISO 14001**（环境） | 追问措施的环境影响（废弃物、排放、能耗），围堵产品的环保处置合规性。 |
| **ISO 45001**（安全） | 追问措施的安全影响，根因中的人因分析需要考虑安全文化和安全屏障。 |

**关键原则**：用户不提标准时，以上全部方法框架仍在运作，但不以"这是 XX 标准第 YY 条要求"的形式呈现——而是以追问的形式呈现。用户提了标准后，明确指出条款号和要求。
